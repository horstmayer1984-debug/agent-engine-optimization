# ヘッドレス SEO: 分離された Web サイトを最適化する方法

ヘッドレス SEO は、分離された Web サイトをクロール可能、インデックス可能、高速かつ理解しやすいものにする手法です。コンテンツまたはコマースのバックエンドがフロントエンドから分離されている場合に適用されます。主なルールは単純です。インターフェイスは、読み込み後にコンテンツを検索するアプリケーションだけでなく、完全な検索可能なドキュメントを提供する必要があります。

##ヘッドレス SEO がカバーする内容

ヘッドレス SEO はヘッドレス CMS SEO よりも広範です。以下が含まれる可能性があります:

- ヘッドレス CMS ウェブサイト
- ヘッドレスショーウィンドウ
- React、Next.js、Nuxt、Astro、または SvelteKit フロントエンド
- コンテンツAPI
- 製品API
- ドキュメントポータル
- AI 読み取り可能なコンテンツ レイヤー

CMS の詳細なガイダンスについては、[ヘッドレス CMS SEO](/es/docs/headless-cms-seo/) を参照してください。取引については、[ヘッドレス取引の説明](/es/docs/headless-commerce-explained/) をお読みください。

##ヘッドレス SEO の優先順位

|優先順位 |何を確認するか |
|
---|
---|
|追跡能力 |重要なページは 200 を返しますが、リンクされておりブロックされていません |
|代表 |メインコンテンツは初期 HTML または信頼できるサーバー出力に存在します。
|メタデータ |ページごとにレンダリングされるタイトル、説明、正規、ロボット、ソーシャル タグ |
|内部リンク |ナビゲーションと本文のリンクは追跡可能なアンカーを使用します。
|構造化データ | JSON-LD は表示されているコンテンツと一致します。
|パフォーマンス |ページは不必要な JavaScript と画像の重みを回避します。
|出版物 |新しいコンテンツはルート、サイトマップ、キャッシュを更新します |
| AI の可読性 |定義、表、FAQ、ソースへのリンクは簡単に抽出できます。

## レンダリング戦略

レンダリング戦略はヘッドレス SEO の中心です。

|戦略 | SEOチューニング |に最適 |
|
---|
---|
---|
|静的生成 |コンテンツを事前に作成できる場合に強い |ブログ、ドキュメント、ランディング ページ |
|サーバーサイドレンダリング |コンテンツが頻繁に変更される場合に強い |価格、入手可能性、パーソナライズされているがインデックス可能なページ |
|増分再生 |頻繁に更新される大規模なサイトに強い |大規模なカタログとコンテンツ センター |
|クライアント側レンダリングのみ |オーガニックランディングページは危険 |接続されたダッシュボードとインデックス不可能なアプリ領域 |

Next.js のドキュメントでは、公式のレンダリング オプションとして静的生成とサーバー側レンダリングについて説明しています。正しい選択は、鮮度、規模、追跡要件によって異なります。

##アプリケーションシェルの問題

多くのヘッドレス サイトは、アプリケーションのシェル (ヘッダー、読み込み状態、およびコンテンツを設定する JavaScript) を誤って送信します。これは最新のブラウザを使用するユーザーには機能する可能性がありますが、SEO や AI 抽出には脆弱です。 Google の [JavaScript SEO Basics] (https://developers.google.com/search/docs/crawling-indexing/javascript/javascript-seo-basics) では、検索は JavaScript を処理しますが、リソースがロックされている場合、ステータス コードが間違っている場合、またはレンダリング後にコンテンツが利用できない場合にはページがクラッシュする可能性があると説明しています。

重要な検索ページの場合は、HTML をハイドレートする前に、HTML を使いやすいものにしてください。

## メタデータ ルール

ヘッドレスのインデックス可能な各ルートには次のものが必要です。

- タイトルタグ
- メタディスクリプション
- H1
- 正規 URL
- 必要に応じてロボット指示
- グラフと Twitter メタデータを開く
- ナビゲーションルートデータ
- 関連する場合のスキーママークアップ

ルート テンプレートが一般的なタイトルや説明を共有することを許可しないでください。これにより、SEO と CTR の両方が低下します。## ヘッドレス SEO と AEO

ヘッドレス アーキテクチャは、コンテンツと機能をビジュアル インターフェイスから分離するため、[エージェント エンジンの最適化](/es/docs/what-is-aeo/) をサポートできます。しかし、エージェントには依然として、公開され、安定した、検証可能なエントリ ポイントが必要です。

ヘッドレス SEO を以下に接続します。

- [AI検索最適化](/es/docs/ai-search-optimization/)

## よくある質問

### ヘッドレス SEO はヘッドレス CMS のみに適用されますか?

いいえ。フロントエンドとバックエンドが分離されている分離されたサイトに適用されます。

### ヘッドレス SEO は従来の SEO よりも優れていますか?

デフォルトのままでは良くありません。これにより、より詳細な制御が可能になりますが、チームは意図的に SEO レイヤーを作成する必要があります。

### ヘッドレス SEO の最大のリスクは何ですか?

ランク付けする必要があるページに対する排他的なクライアントの代表。 2 番目に大きなリスクは、移行中にメタデータが失われることです。

### ヘッドレス SEO は AI 検索に役立つでしょうか?

はい、AI システムが分析および引用できる、構造化され、ソースに裏付けされた、クロール可能なコンテンツをサイトが公開する場合は可能です。
