# Web エージェントにブラウザを使用する

ブラウザの使用状況は、エージェントが実際のブラウザを使用して Web タスクを完了するという急速に成長しているパターンを表すため、AEO にとって重要です。ブラウザ エージェントがタグ、フォーム、エラー、ポリシー、または確認を理解できない場合、ブラウザ エージェントは失敗するか、別のパスを選択します。したがって、AEO はテキストの内容だけでなく、インターフェイスの実際の動作もカバーする必要があります。

## ブラウザの使用状況に注目する価値がある理由

GitHub プラグインは、ブラウザー エージェントの検索で [ブラウザー使用/ブラウザー使用](https://github.com/browser-use/browser-use) として表示されました。 2026 年 6 月 28 日にチェックされた GitHub メタデータには 100,000 を超えるスターが示されており、最も注目されているオープンソース ブラウザ エージェント プロジェクトの 1 つとなっています。

リポジトリの位置付けは単純です。AI エージェントが Web サイトにアクセスできるようにするだけです。これは、[エージェント UX とヒューマンインザループ設計](/es/docs/agent-ux-human-in-the-loop/) および [Lighthouse Agentic Browsing Audit](/es/docs/lighthouse-agentic-browsing-audit/) に直接マッピングされます。

## ブラウザーエージェントが Web サイトに関して明らかにすること

|ウェブサイトの問題 |ブラウザエージェントが苦戦する理由 |
|
---|
---|
|あいまいなボタン | 「続行」または「次へ」では、実際のアクションが表示されない場合があります。 |
|非表示のフォームエラー |エージェントは視覚のみの検証メッセージに気付かない可能性があります。 |
| JavaScript のみのコンテンツ |エージェントは、プレーンな HTML ではなく、レンダリングされた状態を必要とする場合があります。 |
|モーダルとオーバーラップ |タスクのステータスがわかりにくくなる場合があります。 |
|動的な製品データ |エージェントは、安定したバリアントの可用性、価格設定、およびシグナルを必要とします。 |
|弱い確認ページ |エージェントはアクションが完了したことを証明する必要があります。 |

通常、最良の解決策は、機械可読形式で表現された通常の UX 規律です。

## ウェブサイト準備チェックリスト

1. ボタンとフォームフィールドにはわかりやすいラベルを使用します。
2. レンダリング後も、重要なコンテンツを HTML で利用できるようにします。
3. 目に見える、機械が読み取り可能なエラー メッセージを提供します。
4. タスクのエントリ ポイントに安定した URL を追加します。
5. ポリシーを簡単に抽出できるようにします。
6. 購入やアカウントの変更には人間の承認が必要です。
7. 通常の人間のセッションとは別に、エージェントのようなセッションを登録します。

技術情報については、[ヘッドレス ブラウザ オートメーション](/es/docs/headless-browser-automation/) および [Chrome WebMCP オリジン テスト チェックリスト](/es/docs/chrome-webmcp-origin-trial/) を参照してください。

## ブラウザの使用状況と WebMCP の比較

|寸法 |ブラウザの使用スタイル | WebMCP スタイル |
|
---|
---|
---|
|エージェントの対話 |ブラウザからページを操作 |構造化された Web サイト ツールを使用する |
|既存のサイトで動作します |はい |実装された場合のみ |
|信頼性 |ユーザーインターフェイスの明瞭さによって異なります。ツールスキームに依存 |
|最初に最適なソリューション |セマンティック UI と状態を改善する |限定されたアクションを公開する |

どちらの傾向も同じ方向を向いています。Web サイトは、単にクローラーが読み取れるだけでなく、エージェントが使用できる必要があります。

## ブラウザー エージェントの失敗のログを保持し、ページ訪問ごとではなく、タスク ステップごとに失敗を記録します。有用なエントリには、ターゲット アクション、ページ URL、関連する要素またはフィールド、期待される状態、観察された状態、回復の試行、および最終結果が含まれます。これにより、コンテンツの問題とインタラクションの問題が分離されます。インターフェイスに大きな変更を加えるたびに、小さな修正テストを使用します。ポリシーを見つけ、オプションを選択し、有効なフォームを送信し、検証エラーをトリガーし、エラーから回復し、最終状態をコミットします。多くの場合、アクセス可能なラベルによりエージェントの解釈が向上するため、キーボード ナビゲーションを使用して同じタスクを実行します。

文書化された API が成功しているときにブラウザー エージェントが失敗した場合は、その API を優先実行パスとして保持し、ブラウザーの自動化をフォールバックとして扱います。同じビジネス ルールで両方が失敗した場合は、ブラウザ命令を追加する代わりに、基礎となるコントラクトを修正します。

## よくある質問

### ブラウザの使用は SEO ツールですか?

いいえ。これはブラウザ エージェント プロジェクトです。 SEO との関連性は間接的です。発見後にエージェントが Web サイトとどのようにやり取りするかを示します。

### すべてのサイトでブラウザ エージェントをテストする必要がありますか?

フォーム、e コマース、予約、サポート、ダッシュボード、または製品比較フローを備えたサイトでは、少なくとも主要なタスクをテストする必要があります。

### ブラウザの使用は API の代わりになりますか?

いいえ。クリーンな API が存在する場合、エージェントはそれを頻繁に使用する必要があります。ブラウザ エージェントは、Web インターフェイスが利用可能なルートである場合に役立ちます。

### 第一次 AEO テストとは何ですか?

ブラウザー エージェントに安全なタスクを完了するよう依頼し、ページの推測、停止、または誤読の場所を記録します。
