# エージェント取引プロトコルの相互運用性

エージェント取引は単一のプロトコルでは実行されません。 UCP はビジネス システムの相互運用を支援し、AP2 は検証可能な支払い承認を処理し、MCP はツールとコンテキストを公開し、A2A はエージェントを調整し、x402 はマシンネイティブの支払いリクエストを処理し、カード ネットワークの信頼層は ID と支払い制御を追加します。トレンドはプロトコルの独占ではなく、相互運用性です。

## プロトコルの重複が予想される理由

コマースには多くのライフサイクル段階があります。ディスカバリー、カート、チェックアウト、フルフィルメント、返金、認証、エージェントの調整は別の問題です。

単一のプロトコルですべてを解決できると期待すべきではありません。

既存のページ [UCP vs ACP vs MCP](/es/docs/ucp-vs-acp-vs-mcp/) では、3 つの主要なパターンを比較しています。このページは、広範なエージェント取引スタックをマップします。

## プロトコルの役割マップ

|プロトコルまたはレイヤー |主な役割 |取引段階 |
|
---|
---|
---|
| UCP |コマース機能、カート、チェックアウト、購入後の相互運用性 |注文ライフサイクルを通じての発見 |
| AP2 |認可と検証可能な支払い義務 |意図、支払い、支払い |
| PCM |ツールとコンテキストへのアクセス |内部および外部エージェントのワークフロー |
| A2A |エージェント間の調整 |バイヤーエージェント、マーチャント、プラットフォームおよびサービス |
| x402 |ネイティブ HTTP 支払いリクエストと決済パターン | API、コンテンツ、自動支払い |
|検証可能なインテント / TAP / エージェント支払い |信頼、アイデンティティ、トークン化、責任 |承認と支払いのリスク |

## フローの例

代理購入は次のように機能します。

1. エージェントは、ページ、フィード、スキーマ、または UCP カタログ機能を通じて製品を検出します。
2. エージェントは、構造化ポリシーと製品データを使用してオプションを比較します。
3. エージェントは、UCP または Merchant API を介してカートを作成します。
4. ユーザーの意図は、AP2 スタイルのコマンドまたは同等の信頼層を通じて取得されます。
5. 支払いはカード、ウォレット、x402、ステーブルコイン、またはアカウント クレジットを使用して行われます。
6. フルフィルメント イベントにより、支払い後にエージェントが更新されます。
7. 紛争証拠は、ユーザーの意図、エージェントのアクション、カート、支払い、配送を結び付けます。

これは、[エージェント エンジンの最適化](/es/docs/what-is-aeo/) が読み取り層と実行層の両方で作業する必要がある理由を説明しています。

## トレーダーがしてはいけないこと

標準が「勝つ」のを待ってはいけません。代わりに、安定した概念に基づいて内部システムを設計します。

- 製品のアイデンティティ
- ユーザーの意図
- カートのステータス
- 支払い承認
- エージェントのアイデンティティ
- 通商政策
- 注文状況
- 監査証跡

次に、成熟するにつれて、それらの概念をプロトコルにマッピングします。

|

---|
|製品 |スキーマ、フィード、カタログ API、UCP |
|カート | UCP カート API またはビジネス カート |
|意図 | AP2 の義務または検証可能な意図 |
|お支払い | AP2、カードトークン、ウォレット、x402、アカウントクレジット |

## よくある質問

### UCP は AP2 に代わるものですか?いいえ、UCP と AP2 は異なる問題を解決します。 UCP は商用相互運用性に焦点を当てています。 AP2 は、安全な支払いの承認と義務に重点を置いています。

### X402 は AP2 と競合しますか?

場合によっては、同じ支払い会話に表示されることもありますが、それらは異なります。 x402 はネイティブ HTTP 支払いネゴシエーションです。 AP2 は、エージェントの支払いに関する承認と検証可能な意図に関するものです。

### MCP はどこに当てはまりますか?

MCP は、エージェントがツールとコンテキストを使用できるように支援します。ビジネス ワークフローをサポートできますが、チェックアウトやチェックアウト プロトコルに代わるものではありません。

### 販売者は最初に何を実装する必要がありますか?

製品、カート、インテント、チェックアウト、ポリシー、注文、監査などのオブジェクトの内部一貫性から始めます。その後、プロトコルの統合が容易になります。
