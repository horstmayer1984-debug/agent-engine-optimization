#Lighthouse Agentic Navigation 監査ガイド

Chrome Lighthouse エージェント ナビゲーション カテゴリは、従来の開発者監査ワークフローにエージェントの準備状況チェックを組み込むため、重要です。現在のテストは実験的なものですが、Web がどこに向かっているのかを示しています。AI エージェントには、機械可読な概要、検出可能なツール、決定論的な対話面が必要です。

## 何が変わったのでしょうか?

Chrome for Developers では、エージェントのナビゲーションに関する Lighthouse 監査が文書化されるようになりました。文書化されたチェックには、「llms.txt」監査と、登録された WebMCP ツールの情報監査が含まれます。 Chrome は、このカテゴリは実験的なものであり、提案された標準に基づいているため、最終スコアではなく初期の診断レイヤーとして扱う必要があると述べています。

一次情報源:

- [Chrome: Lighthouse 監査 llms.txt](https://developer.chrome.com/docs/lighthouse/agentic-browsing/llms-txt)
- [Chrome: Lighthouse エージェント ナビゲーション スコア](https://developer.chrome.com/docs/lighthouse/agentic-browsing/scoring)
- [Chrome: 登録された WebMCP ツールの監査](https://developer.chrome.com/docs/lighthouse/agentic-browsing/registered-webmcp-tools)

## 監査で証明されること

|監査 |意味 | AEO の実践的な回答 |
|
---|
---|
---|
| llms.txt |サイトのルートが機械可読な概要を提供しているかどうかを確認します。重要なリンクを含む簡潔な `llms.txt` を投稿します。
|登録済み WebMCP ツール |ページがブラウザ エージェントに公開するツールをリストします。明確なツール名とアクションの説明を追加します。
|エージェントのナビゲーション カテゴリ |承認/拒否シグナルと情報準備を表示 |ランキングスコアではなく、技術的なチェックリストとして使用 |

Chrome の `llms.txt` ドキュメントでは重要な違いが示されています。欠落しているファイルはオプションであるため、「該当なし」とマークされる場合がありますが、サーバー エラーにはフラグが立てられます。つまり、「llms.txt」が壊れているということは、ファイルがまったくないよりも悪いということです。

## これが AEO にとって重要な理由

[AEO Readiness Audit](/es/docs/audit/) は、検出、読み取りレイヤー、実行レイヤー、および信頼をすでに検証しています。 Lighthouse は、開発者が使い慣れたツール環境内でこれらの信号の一部をキャプチャする簡単な方法を追加します。

これは、完全な AEO 監査に代わるものではありません。開発者が特定の技術部分を検証するのに役立ちます。

- サイトのルートは「llms.txt」を適切に提供しますか?
- ページ アクションは WebMCP ツールとして表示されますか?
- ブラウザのテスト中に、実験的なエージェントの準備状況のシグナルが表示されますか?

## llms.txtの準備方法

エージェントのナビゲーションには、「llms.txt」は短くて便利である必要があります。サイトマップのクローンではありません。

実用的なファイルには以下を含める必要があります。

- サイトの目的
- 基本的な概念
- 最高のホームページ
- 主要なツールまたはサービス
- 機械可読リソース
- 更新日

[llms.txt ガイド](/es/docs/programming-llms-txt/) では、その構造化方法について説明しています。主なルールはシンプルです。エージェントが最初に何を読むべきかを理解できるようにすることです。

## WebMCP ツールを準備する方法登録された WebMCP ツールの監査は有益です。登録されたツールがない場合、リストは空になります。これは自動的に失敗するわけではありませんが、ページがブラウザ エージェントにとって使いやすい方法でアクションを公開しているかどうかを明らかにします。

最初に適したツールには次のものがあります。

- 監査を要求する
- 予約をする
- 見積もりを依頼する
- カートに追加
- 空き状況を確認する
- サポートチケットを送信する
- 製品を比較する

[WebMCP ガイド](/es/docs/webmcp-agent-ready-websites/) では、ページ レベルのツールとサーバー側 MCP をいつ使用するかについて説明しています。

## 推奨されるワークフロー|ステップ |アクション |出力 |
|
---|
---|
---|
| 1 |ホームページと重要なテンプレートで Lighthouse を実行する |ナビゲーション サイン紹介エージェント |
| 2 | `llms.txt` 応答エラーを修正 |安定した機械可読ルートの概要 |
| 3 | WebMCP を低リスク アクション ページに追加する |登録されたツールが表示されます |
| 4 |灯台を繰り返す |検出を確認する |
| 5 |エージェントアクションのサーバー側ログを追加する |測定層 |

このワークフローは、技術的検証を [実行レイヤー](/es/docs/execution-layer/) に接続します。監査に合格することは、エージェントが有用なタスクを完了できることを証明することほど重要ではありません。

## してはいけないこと

チェックに合格するためだけに偽の「llms.txt」ファイルを追加しないでください。 「submit_form」のような汎用 WebMCP ツールを意味のある説明なしに登録しないでください。監査が存在するため、リスクの高いアクションを公開しないでください。

最良のエージェント ナビゲーションの実装は、明確なコンテンツ、安定したアクション、明示的な権限、適切なログなど、退屈なものです。

## よくある質問

### Lighthouse エージェントのナビゲーション スコアはランキング要素ですか?

公式情報筋はそんなことは言っていない。これは、Google のランキング要素ではなく、実験の準備が整っていることの兆候として扱います。

### llms.txt は必要ですか?

Chrome のドキュメントには、現時点ではオプションであると記載されています。欠落しているファイルは「該当なし」となる場合がありますが、サーバー エラーにフラグが立てられる場合があります。

### WebMCP を各フォームに追加する必要がありますか?

いいえ。エージェントの実行がユーザーに役立つ、価値が高くリスクの低いアクションから始めます。

### これは通常の SEO 監査とどう違うのですか?

従来の SEO 監査は、クロール可能性、速度、メタデータ、コンテンツに焦点を当てていました。エージェントのナビゲーション監査は、エージェントがサイトを理解し、サイト上で行動できるかどうかに焦点を当てます。

### 実装後に何を測定する必要がありますか?

エージェントによってトリガーされたアクション、フォーム入力、エラー、無効なリクエスト、およびエージェントが行動する前に読むページを追跡します。
