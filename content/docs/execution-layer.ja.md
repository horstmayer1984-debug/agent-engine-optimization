AI システムが生成された応答でそのコンテンツを引用するとき、それが作業中の読み取りレイヤーになります。役に立ちますが、消極的です。

AI エージェントが API を使用してリアルタイムの在庫を確認し、ユーザーの予算制約と比較して購入を完了するとき、これが実行層です。アクティブでトランザクション的であり、インフラストラクチャに必要なものが根本的に異なります。

実行層は、AEO がそれまでのすべてから分離される場所です。これはコンテンツの最適化の問題ではありません。これは **インフラストラクチャ アーキテクチャの問題**です。

---

## どのエージェントが行動する必要があるか

タスク (来週の火曜日にバルセロナで 150 ユーロ未満のホテルの部屋を予約する) を実行するフリーランス エージェントには、次のものが必要です。

### 1. 発見

タスクの種類に一致するサービスを見つけます。これは部分的には読み取り層の問題です。構造化データ、スキーマ マークアップ、および適切なインデックス作成は、エージェントがサービスを発見するのに役立ちます。しかし、発見だけでは行動はできません。

### 2. 容量を理解する

サービスで何ができるのか、どのような制限があるのかを把握してください。エージェントには、どのようなアクションが利用可能か、どのようなパラメータが受け入れられるか、どのような結果が返されるか、どのような制限が適用されるかなど、その機能の機械可読な記述が必要です。これは、MCP ツールの機能マニフェストと説明が提供するものです。

### 3. リアルタイム データへのアクセス

現在の価格、在庫状況、条件を確認します。昨日の価格を含む静的 HTML ページは機能しません。エージェントには、プログラムで評価できる現在の構造化データを返すエンドポイントが必要です。

### 4. 制約のマッチング

オファーがユーザーの要件に適合するかどうかを評価します。予算は 150 ユーロ未満ですか?火曜日は利用可能ですか？バルセロナで？朝食は含まれていますか？エージェントは、これらの制約をそのデータと比較する必要があります。これには、マーケティングコピーではなく、構造化された決定論的なデータが必要です。

### 5. アクションの実行

予約、購入、またはリクエストを有効にします。エージェントにはトランザクション エンドポイントが必要です。ここで、UCP (取引用) や MCP (一般ツール アクセス用) などのプロトコルが登場します。

### 6. ステータスの確認

アクションが成功したことを確認します。予約は済んでいますか？エージェントには、利用可能、予約済み、確認済みなどの観察可能な状態遷移が必要です。検証がなければ、エージェントは信頼性の高いレポートを作成できません。

## ビルディングブロック

### アクションスキーム

Schema.org では、BuyAction、ReserveAction、OrderAction などのアクション タイプを定義します。これらは、ページ上またはサービスを通じてどのようなアクションが可能であるかをエージェントに伝えます。ほとんどの Web サイトは、情報マークアップのみに Schema.org を使用します。これをアクションレベルのマークアップに拡張することが、バックエンド API を必要としない最初のステップです。

### 機能マニフェストサービスができること、サービスがサポートするプロトコル、およびサービスと対話する方法を説明する機械可読文書。これは、エージェントの機能を表す robots.txt ファイルのようなものだと考えてください。ただし、何を追跡できるかを示すのではなく、何ができるかを示す点が異なります。

### 確定的な入力と出力

マーケティング ページは人間を説得するように設計されています。エージェント向けデータは、固定パラメーター、構造化された応答形式、明確な値、機械が読み取り可能なエラー コードなど、機械による意思決定を考慮して設計する必要があります。

ウェブサイトの内容:

> 「当社のプレミアム プランは、チームが必要とするすべての機能を備え、月額わずか 49 ドルから始まります。」

エージェントに必要なもの:

```json
{
  "plan": "premium",
  "price": { "amount": 49, "currency": "USD", "interval": "month" },
  "features": ["feature_a", "feature_b", "feature_c"],
  "limits": { "users": 50, "storage_gb": 100 },
  "trial": { "available": true, "days": 14 }
}
```

### プロトコルの互換性**MCP (Model Context Protocol)**: AI モデルが標準化されたインターフェイスを通じて外部ツールやデータにアクセスできるようにします。 MCP サーバーを備えた販売者は、MCP 対応エージェントがカスタム統合なしでカタログをクエリできるようにします。

**UCP (Universal Commerce Protocol)** – すべてのエージェントが同じプロトコルを使用して任意の販売者と取引できるように、購入プロセス全体を標準化します。

**A2A (エージェント間)**: エージェントが専門のエージェントにタスクを委任できるようにします。

これらのプロトコルを早期に実装した組織は、エージェントのタスクを完了するためのデフォルトのオプションになります。

### 状態遷移

エージェントは、アクションが期待どおりの結果を生み出すことを確認する必要があります。これには、観察可能な状態遷移が必要です。

`利用可能 > 予約済み > 支払い保留中 > 確認済み > 履行済み`

各遷移はクエリ可能である必要があります。これがないと、エージェントは複数ステップのワークフローを確実に管理できません。

### 信頼と検証

2,000 ユーロの旅行を予約するエージェントは、情報源を信頼する必要があります。エージェントの信頼シグナルは、人間の信頼シグナルとは異なります。エージェントには、検証可能な販売者 ID、暗号署名 (AP2) 取引受領書、すべてのタッチポイントにわたる一貫性のある正確なデータ、透明性のある制限文書、および監査証跡が必要です。

## これがあなたの組織にとって何を意味するか

実行層は一夜にして構築されるものではありません。それは段階的に開発される能力です。

**フェーズ 1 (現在):** エージェントの現在の準備状況を監査します。構造化データ、スキーマ カバレッジ、コンテンツ構造、および既存の API フットプリントを評価します。

**フェーズ 2 (3 ～ 6 か月):** 読み取りレイヤーの最適化を実装し、実行レイヤーのインフラストラクチャの計画を開始します。アクション スキーマの追加、機能ドキュメントの作成、プロトコルの互換性の評価。

**フェーズ 3 (6 ～ 18 か月):** 実行層の機能を開発します。 API の公開、プロトコル サポートの実装、エージェント向けデータ エンドポイントの作成、ヘルス チェック メカニズムの開発。[AEO 準備監査](/es/docs/audit/) – 体系的な評価から始めます。

[AEO とは?](/es/docs/what-is-aeo/): 完全な定義と範囲。

## よくある質問

### ランタイム層は Web サイトのフロントエンドに置き換わりますか?

いいえ、ヒューマン インターフェイスは依然として重要です。実行層は、ソフトウェアがビジュアル インターフェイスがどのように動作するかを推測することなく使用できる構造化されたパスを追加します。

### エージェントにとってアクションが安全なのはなぜですか?

アクションには、明示的な入力、検証、承認、安定した状態、制限された再試行、明確なエラー、および検証可能な結果が必要です。

### チームはすべての内部アクションを公開する必要がありますか?

いいえ。定義されたユーザー特典、権限モデル、所有者、監査ログ、およびフェイルオーバー プロセスを持つアクションのみを公開します。
