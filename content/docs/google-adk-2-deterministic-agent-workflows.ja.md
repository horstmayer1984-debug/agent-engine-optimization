# Google ADK 2.0 と決定論的なエージェントのワークフロー

Google ADK 2.0 は、パスが既知で監視可能、回復可能な決定論的なワークフローを目指してエージェント設計を推進するため、AEO にとって重要です。それはまさに、エージェントがページを読んでから予約、購入、サポート リクエスト、ビジネス プロセスを完了するまでに Web サイトが必要とするものです。

## Google が発表した内容

Google は、2026 年 7 月 1 日に [ADK 2.0 を構築した理由](https://developers.googleblog.com/why-we-built-adk-20/) を公開しました。中心的な議論は単純です。大規模な言語モデルは推論に役立ちますが、ワークフローが明確な場合、実稼働エージェントも決定論的な実行を必要とします。

Google はまた、グラフベースのワークフロー エンジン、組み込みの人的サポート、動的オーケストレーション、再試行、状態、および再開動作を備えた [ADK Go 2.0](https://developers.googleblog.com/announcing-adk-go-20/) も発表しました。

## 決定論的なワークフローが重要な理由

多くの Web タスクは即興で行うべきではありません。返金、支払い、見積もりリクエスト、保険請求、またはサポートのエスカレーションには、予測可能なパスが必要です。

|エージェントの動作 |良いこと |リスク |
|
---|
---|
---|
|自由形式の推論 |調査、比較、要約 |実行パスが不明瞭 |
|ツールコール |データの取得またはステップの実行 |制限のないツールの悪用 |
|決定的なワークフロー |既知の複数ステップのプロセス |初期モデリングが必要 |
|ループ内の人間のワークフロー |危険な行為または取り返しのつかない行為 |遅いけど安全 |

[実行層](/es/docs/execution-layer/) は、重要なビジネス アクションに対して決定論的なパスを優先する必要があります。

## Web サイトに対する AEO の影響

ADK 2.0 は、エージェントにボタンやマーケティング コピーからワークフローを推測させないという経験則を強化します。ワークフローを公開します。

つまり、次のように定義します。

- 許可されるタスクの状態
- チケットが必要です
- 検証ルール
- 承認ゲート
- 再試行ロジック
- キャンセルルート
- 最終確認メッセージ
- 監査ログ

これは、[エージェント UX とヒューマンインザループ設計](/es/docs/agent-ux-human-in-the-loop/)、[マルチエージェント AEO](/es/docs/multi-agent-aeo/)、および [エージェント対応 Web アプリ](/es/docs/agent-ready-web-apps/) に接続します。

## 例: 予約の予約

|ステップ |決定論的な要件 |
|
---|
---|
|サービスを選択 |ページ テキストだけでなく、安定したサービス ID を使用してください。
|空室状況を確認する |現在のタイムスロットを返す |
|ユーザーデータを収集する |必須フィールドを検証する |
|ポリシーを表示 |返品の価格とキャンセルのルール |
|予約を確認する |ユーザーの明示的な承認が必要 |
|結果を返す |予約 ID と次のステップを入力してください |

エージェントは、どのサービスがユーザーに適しているかを推論できます。予約確認ルートを発明すべきではありません。

## サイトを監査する方法

1. 価値の高いワークフローを選択します。
2. そのワークフローの正確なステート マシンを作成します。3. Web サイトが現在視覚的な提案に依存している部分を特定します。
4. 必要に応じて、機械可読タグ、スキーマ、API、または MCP ツールを追加します。
5. 取り消し不能なアクションの前に承認ゲートを追加します。
6. エージェントに関連する各状態遷移を記録します。
7. 成功の道だけでなく、失敗の道も試してください。

## よくある質問

### 決定論的なワークフロー設計は開発者だけを対象としたものですか?

いいえ。製品、SEO、法務、サポート、運用チームは、どのアクションに承認が必要か、どのステータスが有効かを定義するのに役立つ必要があります。### ADK 2.0 は LangGraph または他のフレームワークを置き換えますか?

いいえ、これはエージェント開発の別の道です。 AEO からの有益な教訓は、既知のワークフローの決定論的な実行という設計原則です。

### これが SEO に影響を与えるのはなぜですか?

クラシック SEO はユーザーをページに誘導します。 SEO エージェントは、自動アシスタントがそれらのページとワークフローを使用できるようにすることも必要です。

### 最初にモデル化するワークフローは何ですか?

予約、見積もりリクエスト、製品比較、支払い、サポート チケット、デモ リクエストなど、エージェントが合理的に実行できる最も価値の高いアクションから始めます。
