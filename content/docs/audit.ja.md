デジタル インフラストラクチャを 2 つの層で体系的に評価します。

**読み取りレイヤー**: AI システムはコンテンツを検出して理解できますか?

**実行層**: 自律エージェントはインフラストラクチャを通じて動作できますか?

ほとんどの組織は、読書層でかなり良いスコアを獲得しています。実行層の準備状況を評価した人はほとんどいません。

---

## 監査の範囲

### レイヤー評価の読み取り

**構造化されたデータ範囲**: Schema.org マークアップの整合性と正確性。ページとプラットフォーム間のエンティティの一貫性。 JSON-LD実装の品質。

**AI のコンテンツ構造**: セマンティック HTML の品質。情報密度とマーケティング散文の比率。よくある質問と質疑応答の構造。コンテンツの新鮮さの兆候。

**AI の発見可能性**: llms.txt の存在と品質。 AI クローラー用に robots.txt を構成します。サイトマップの構造。クロスプラットフォームの一貫性。

**権限標識** — E-E-A-T インジケーター。出典の帰属。外部検証。

### 実行層の評価

**API Surface**: 既存の API の可用性とドキュメントの品質。 OpenAPI仕様の存在。エージェントのエンドポイント検出。応答フォーマットの構造。

**プロトコルの準備**: MCP 互換性評価。 UCP（貿易用）の準備。既存の統合パターン。マシンにアクセスするための認証メカニズム。

**データ決定論**: 製品およびサービスの構造化データと非構造化データ。リアルタイムのデータ可用性。機械可読な制限。すべてのタッチポイントにわたる一貫したデータ。

**アクション性**: 利用可能な取引タイプ (購入、予約、リクエスト、購読)。状態遷移モデリング。マシンコンシューマのエラー処理。検証メカニズム。

**信頼できるインフラストラクチャ**: 企業の ID 検証。データの出所と正確性。コンプライアンスの準備 (GDPR、EU AI 法、該当する場合)。監査証跡機能。

## 成果物

各監査では以下が生成されます。

1. **AEO 準備スコア**: 業界と比較した読み取りレイヤーと実行レイヤーの準備状況の数値評価。

2. **ギャップ分析**: 何が欠けているのか、何が機能していないのか、何がうまく機能しているのかを優先順位付けした具体的なリスト。

3. **即効性**: 変更は数日で実装可能で、エージェントの準備に目に見える影響を与えます。

4. **戦略的ロードマップ**: 業界のエージェント導入曲線に合わせてスケジュールを調整した、実行層機能を構築するための段階的な計画。

5. **プロトコル互換性レポート** – どのプロトコル (MCP、UCP、A2A) が貴社のビジネスに関連しており、どのような実装が必要であるかを評価します。

## 監査形式

### AEO Readiness Audit - Lightweight 実行層の概要とともに読み取り層に焦点を当てています。

**範囲:** Web サイト、一次デジタル資産。
**納期:** 5営業日。
**最適な対象:** AEO、主にコンテンツベースのビジネスを検討し始めている組織。

### AEO 準備状況監査 – 完了

API インフラストラクチャとプロトコルの互換性分析を含む、両方のレイヤーの包括的な評価。**範囲:** Web サイト、API、データ インフラストラクチャ、マーケットプレイスの統合。
**納期:** 10営業日。
**次の用途に最適:** E コマース、SaaS、旅行と予約、および既存の API インフラストラクチャを使用するあらゆるビジネス。

## なぜ今?

プロトコルはライブです。 UCP は 2026 年 1 月に発売されました。MCP は運用の準備ができています。エージェント主導のコマースは測定可能であり、成長しています。

評価と準備を行っている組織は、現在、複合的なメリットを享受しています。エージェントのトラフィックが大量になるまで待った場合、よりコストがかかり、破壊的な最新化に直面することになります。

## 再現可能な監査のための証拠規則

それぞれの結果には、テストされた URL またはエンドポイント、リクエスト方法、観察された結果、予想される結果、およびユーザーまたはビジネスへの影響に基づく重大度が含まれている必要があります。スクリーンショットは視覚的な欠陥には役立ちますが、マシンが直面する問題には生の HTML、応答本文、ヘッダー、またはスキーマ検証結果も必要です。

補正前と補正後は同じサンプルセットを使用してください。実際のサンプルには、ピラー ページ、比較ページ、トランザクション ページ、API アクション、既知のエラー パスが含まれています。これにより、変更後の単純なページのテストによって引き起こされる誤った改善が防止されます。

重大度は結果を反映する必要があります。オプションの説明が欠落している場合、優先度は低くなります。古い価格、あいまいな承認ステータス、または誤って繰り返される可能性のあるアクションは、エージェントが間違った決定を下したり、重複したトランザクションを作成したりする可能性があるため、優先度が高くなります。

## よくある質問

### AEO 監査では最初に何をテストする必要がありますか?

正規ページ、構造化コンテンツ、重要なユーザー タスク、状態変化を引き起こすインターフェイスから始めます。成功したルートと失敗したルートをテストします。

### AEO 監査は単なるコンテンツレビューですか?

いいえ、コンテンツは読書層の一部です。完全な監査では、API、プロトコル調整、認可、決定論的データ、エラー、確認、監査可能性も検証します。

### 監査はどのくらいの頻度で繰り返す必要がありますか?

マテリアルのコンテンツ、スキーマ、API、ポリシー、またはワークフローを変更した後、特定のテストを繰り返します。エージェントのトラフィックとサポートされる機能が進化するにつれて、より大きなサンプルを定期的に確認します。
