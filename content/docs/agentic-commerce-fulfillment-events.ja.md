# エージェント取引履行イベント

代理取引は支払いが承認されても終了しません。エージェントは、注文ステータス、出荷イベント、配達確認、キャンセル、返品、返金、保証およびロイヤルティ データも必要とします。購入後のステータスが機械で読み取れない場合、エージェントは支払い後にタスクを処理できません。

## コンプライアンスが AEO の一部である理由

従来の電子商取引では、フルフィルメントを管理プロセスとして扱うことがよくあります。エージェント取引は、エージェントが注文の監視を担当できるため、ユーザー向けのワークフローの一部になります。

たとえば、エージェントは次のことを行う必要がある場合があります。

- タスクを閉じる前に配信を確認する
- コンプライアンスに失敗した場合は再注文します
- 返金をリクエストする
- 遅延についてユーザーに最新情報を伝える
- ロイヤルティ特典を適用する
- サブスクリプションが開始されていることを確認します

これは実行層の動作です。 【エージェントトレードの実行】(/es/docs/agentic-commerce-execution/)、【エージェントトレードの5段階】(/es/docs/five-levels-agentic-commerce/)と併せて属します。

## エージェントが必要とする購入後のイベント

|イベント |エージェントの使用 |
|
---|
---|
|注文受付中 |購入が存在することを確認します |
|支払いが捕捉されました |お金の動きを確認 |
|予約在庫 |キャンセルのリスクを軽減 |
|提出物が作成されました |追跡の可視性を提供します |
|配達確認済み |タスクを完了としてマークする |
|返品開始 |例外処理の開始 |
|払い戻しが行われました |解像度を確認する |
|ロイヤルティが適用される |特典と ID リンクを確認する |

## UCP と注文ライフサイクルの考え方

UCP のドキュメントでは、購入後のエクスペリエンスを含む、発見と支払いを超えたコマース機能について説明しています。 Google はまた、エージェント コマースの過程に沿ったインフラストラクチャとして UCP を構築します。

エージェントは領収書以上のものを必要とするため、これは重要です。明確な意味を持つ状態遷移が必要です。

## 導入チェックリスト

1. 安定した注文 ID とカート ID を使用します。
2. 支払い後に構造化確認書を返送します。
3. 追跡ステータスを機械可読形式で表示します。
4. キャンセルおよび返品の資格規則を公開します。
5. 返金ステータスと返品ステータスを分けてください。
6. 承認されたら、ロイヤルティ ID を注文に接続します。
7. サポート チームのエージェント開始順序イベントをログに記録します。
8. エージェントがユーザーに説明できるエラー状態を提供します。

[ユニバーサル カート ガイド](/es/docs/universal-cart-agentic-commerce/) は前払い側について説明しています。このページでは支払い後の内容について説明します。

|リスク |影響 |修正 |
|
---|
---|
---|
|曖昧な順序状態 |エージェントは何が起こったのか説明できません。明示的なステータス コードを使用する |

## フルフィルメント イベントを安定したコントラクトとして定義する各フルフィルメント イベントは、注文、イベント タイプ、タイムスタンプ、現在の状態、前の状態、および次に許可されるアクションを識別する必要があります。 Webhook の繰り返しによって重複した払い戻し、キャンセル、ロイヤルティの調整が発生しないように、イベントには安定した識別子も必要です。

人間が判読できるステータス テキストを、契約ではなくプレゼンテーションとして扱います。エージェントは、「出荷済み」、「配達済み」、「返品要求済み」、または「返金済み」などの文書化された状態で行動する必要があります。 「途中」などの自由テキストは顧客メッセージに残すことができますが、ワークフロー ロジックを制御すべきではありません。

発売前に 3 つの失敗ケースをテストします。イベントが順序どおりに到着しない場合、同じイベントが 2 回到着する場合、および注文全体が終了しない部分的な払い戻しです。信頼できる実装では、履歴が保存され、各インスタンスの後に現在の権限状態が公開されます。

## よくある質問

### エージェントがコンプライアンス イベントを必要とするのはなぜですか?

委任されたタスクの多くには、注文するだけでなく結果の追跡も含まれるためです。

### コンプライアンスは UCP の一部ですか?

UCP は、購入後のエクスペリエンスを含む、チェックアウトを超えたコマース フロー向けに設計されています。実装の詳細は販売者とプラットフォームによって異なります。

### トレーダーは最初に何を公開する必要がありますか?

注文の確認、配送状況、配達状況、キャンセルの資格、返品の資格、返金の状況。

### これは SEO に影響しますか?

AEOに影響します。エージェントはタスクを安全に完了し、ユーザーに情報を提供し続けるために、信頼できる購入後データを必要とします。
