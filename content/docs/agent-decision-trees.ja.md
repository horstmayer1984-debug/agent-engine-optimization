AI エージェントは人間のように Web サイトを評価しません。彼らはナビゲートすることはなく、印象を形成することもなく、少しずつ信頼を築いていきます。各ノードで厳密な合否基準を備えた決定論的なデシジョン ツリーを実行します。検証が失敗した場合、プロセスは終了します。エージェントは説明なしに次のプロバイダーに進みます。

これらの決定木を理解することで、選択されるか、黙って除外されるかの違いが生じます。

## すべてのエージェントが尋ねる 6 つの質問

### 質問 1: このシステムで何ができるのですか?

エージェントは能力の明示を求めます。 UCP 実装では、これは /.well-known/ucp 内の JSON ファイルです。 MCP 実装では、これらはサーバーから返されるツールとリソースのスキーマです。より単純な構成では、llms.txt ファイルまたはエージェント カードになります。

エージェントがシステムが何をサポートしているかを判断できない場合は、それを完全に無視します。部分的なクレジットはありません。機能が宣言されているか、宣言されていないかのどちらかです。

承認条件: サポートされているアクション、データ型、およびトランザクション機能に関する機械可読なステートメント。

エラー条件: マニフェスト、スキーマ、llms.txt がありません。エージェントには、あなたが何を提供するかを判断する方法はありません。

### 質問 2: どのような備品が必要ですか?

エージェントができることを理解したら、アクションをトリガーするために必要な入力があるかどうかを確認します。システムは、データ型、必須フィールドとオプションフィールド、検証ルール、および許容範囲など、予期される入力値の明確な概要を提供する必要があります。

合格条件: 各エンドポイントまたはアクションには、タイプ、制約、例を含む文書化された入力スキーマがあります。

エラー状態: エントリー要件があいまいで文書化されていない、または試行錯誤によってのみ発見できる。

### 質問 3: 現在の状態は検証可能ですか?

エージェントは静的ページのコンテンツを信頼しません。リアルタイムで重大なステータスを確認する必要があります。

価格は明示的かつ最新のものですか? 「要求に応じた価格」や「現時点で」などの人間的なフレーズは、リスク回避取引で即座に排除される原因となります。エージェントは、定義された期間に有効な正確な番号を必要とします。

在庫はリアルタイムで確認されますか?エージェントは API 呼び出しを送信します。実際の在庫が 2 時間前に売り切れていた場合、キャッシュされた「在庫あり」という HTML は価値がありません。

ポリシーは構造化されており、機械で読み取り可能ですか?返品ポリシー、配送条件、キャンセル条件。法律上のページに散文の段落としてのみ存在する場合、エージェントはそれらを確実に分析できません。 Schema.org アクション タグと構造化されたポリシー ステートメントが必要です。

合格条件: 各重要なトランザクション パラメータの検証可能なリアルタイム ステータス。障害状態: 古いデータ、あいまいな価格設定、構造化されていないポリシー、または分析ではなく解釈が必要なパラメータ。

### 質問 4: アクションの後にどのような状態が保証されますか?

エージェントには、動作時に何が起こるかについての決定的な保証が必要です。購入を送信すると、注文 ID が記載された確認を受け取りますか?予約をした場合、確認された時間枠を受け取りますか?アクションが失敗した場合、その理由を説明する構造化エラーが表示されますか?

合格条件: 各アクション エンドポイントは決定的な結果 (コミットの詳細で成功、または特定のエラー分類で失敗) を返します。

エラー状態: あいまいな応答、構造化データなしの人間の言語による確認、または検証可能な証拠なしで成功を返すエンドポイント。

### 質問 5: 属性はソース間で一致しますか?エージェントは複数のソースからのデータを相互参照します。 Google Merchant Center の価格が API レスポンスと異なる場合、またはスキーマ マークアップが UCP マニフェストとは異なる在庫状況を示している場合、エージェントはその不一致を検出します。

矛盾は混乱を引き起こすだけではありません。それらは排除を引き起こします。あなたのデータを信頼できないエージェントは、あなたのシステムを通じて取引を行うことはありません。

承認条件: すべての機械可読ソース (スキーマ マークアップ、API 応答、製品フィード、UCP マニフェスト) における価格、可用性、仕様、およびポリシーの同一の値。

失敗条件: データ ソース間の不一致。

### 質問 6: アクションを安全に繰り返すことができますか?

エラーが発生した場合、エージェントは再試行します。購入後にネットワークがタイムアウトすると、エージェントは同じリクエストを再度送信します。システムは、重複した注文、重複した請求、または破損したステータスを作成することなく、これを処理する必要があります。

合格条件: 同じリクエストが何回送信されたかに関係なく、同じ結果が生成される冪等のトランザクション エンドポイント。

エラー状態: 再試行により重複が作成される非冪等エンドポイント。

## 実際のデシジョンツリー

エージェントはこれらの質問を順番に評価します。 「失敗」が発生するたびに評価が直ちに終了します。

宣言された容量?いいえ→除外します。文書化された入力?いいえ→除外します。リアルタイムでステータスを確認可能?いいえ→除外します。決定的な結果?いいえ→除外します。データはソース間で一貫していますか?いいえ→除外します。安全な再試行?いいえ→除外します。

6 つのチェックすべてに合格した場合にのみ、エージェントはトランザクションを続行します。

## これが人間の評価と異なる理由人間の購入者は曖昧さを許容します。 「リクエストに応じて価格」とは、電話で問い合わせることを意味します。 「通常3～5日で発送」で十分近いです。あなたは、わかりにくい返品ポリシーを閲覧し、同意します。

エージェントはこのようなことを一切許しません。すべてのあいまいさは失敗の条件です。 「通常」はすべて不定です。構造化されていないポリシーは分析できません。エージェントは、検証または除外という二項論理で動作します。

人間の目には洗練されているように見えるサイトでも、エージェントからはスコアが 0 になる可能性があるのはこのためです。ビジュアルデザイン、ブランドの評判、説得力のあるコピーは、決定論的な意思決定ツリーでは重要ではありません。

[実行層ガイド](/es/docs/execution-layer/) では、インフラストラクチャ要件について説明しています。 [AEO 実装ガイド](/es/docs/implement-aeo/) では、段階的な最適化パスについて説明しています。

## よくある質問

**AEO の決定論的ディシジョン ツリーとは何ですか?**
エージェントがサイトと取引する前に実行する承認/失敗チェックの厳密なシーケンス。各検証では、特定の要件 (機能、入力、状態、結果、一貫性、べき等) が評価されます。検証に失敗すると、即時に除外されます。

**エージェントがサイトを除外する最も一般的な理由は何ですか?**
構造化されていない、または曖昧な価格。 「リクエスト時の価格」、「現在の価格」、またはスキーマ マークアップと API 応答の間で価格が異なる場合、即時除外が発生します。

**サイトがエージェントの評価に合格したかどうかをテストするにはどうすればよいですか?**
6 つの質問すべてを手動でシミュレートします。自分の能力に関する記述は見つかりましたか?すべてのエントリは文書化されていますか? API呼び出しを使用してステータスを確認できますか?すべてのデータ ソースは同じ値を返しますか?このチェックは毎月実行します。**機能マニフェストとは何ですか?**
システムが何ができるかを宣言する機械可読ファイル。 UCP の場合: /.known/ucp。 MCP: ツールとリソースのスキーム。より単純な構成の場合: llms.txt または Agent-card.json。

**冪等性がそれほど重要なのはなぜですか?**
エージェントが失敗しても再試行するためです。タイムアウトは、アクションが失敗したことを意味するものではありません。エンドポイントが再試行時に重複した注文を作成した場合、エージェントは実際の損害を引き起こしています。べき等エンドポイントは、同じリクエストに対して同じ結果を返すことにより、再試行を安全に処理します。
