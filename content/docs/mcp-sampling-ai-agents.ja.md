# MCP サンプリングの説明: サーバーが AI モデル出力をリクエストする方法

MCP サンプリングを使用すると、MCP サーバーはワークフロー中にモデル結果を生成するようにクライアントに要求できます。サーバーには独自のモデル API キーは必要ありません。クライアントは、モデルのアクセス、権限、レビュー、および最終的な配信の制御を維持します。これにより、サンプリングはエージェント ワークフローにとって便利になりますが、生成されたプロンプトと結果がユーザーに明確に表示されない場合は危険です。

## MCP サンプリングとは何ですか?

公式の [モデル コンテキスト プロトコル サンプリング仕様] (https://modelcontextprotocol.io/docs/concepts/sampling) では、サーバーがクライアント経由で言語モデルから LLM 生成を要求する方法としてサンプリングについて説明しています。プロトコル リビジョン 2025-06-18 では、それをサポートするクライアントは「sampling」機能を宣言し、サーバーは「sampling/createMessage」を介して生成を要求します。

抽象的に聞こえるので、簡単に説明します。ツール サーバーは、ユーザーの AI クライアントに、ツール フローの一部としてテキストの思考、要約、並べ替え、生成を依頼できます。

より広範なコンテキストについては、[MCP とエージェント API](/es/docs/mcp-vs-api-for-agents/)、[OAuth による MCP 認証](/es/docs/mcp-authorization-oauth-ai-agents/)、および [MCP アプリケーション](/es/docs/mcp-apps-guide/) を参照してください。

## サンプリングが存在する理由

サンプリングを行わない場合、MCP サーバーには次の 2 つの限られたオプションがあります。

- 生データを返し、クライアント モデルが何をすべきかを知るのを待ちます
- 独自のモデルプロバイダーに直接電話をかけると、コスト、プライバシー、キー管理の問題が発生します

サンプリングにより中間パスが作成されます。サーバーはモデル作業を要求できますが、クライアントは依然としてゲートキーパーです。

## 使用例の例

サンプリングは、サーバーが言語モデルの結果を必要とするが、モデルの関係を所有すべきではない場合に役立ちます。

役立つパターンは次のとおりです。

- 回収した文書を要約する
- ツールがデータをフェッチした後に応答を作成します
- サポートチケットを分類する
- 構造化データから自然言語による説明を生成する
- ユーザーが提供したコンテキストを変換するようにクライアント モデルに依頼します。
- ユーザーに見える基準に基づくスコアリングオプション AEO の場合、Web サイトとエージェントが読み取り可能な API は、検索、実行、説明を組み合わせたワークフローをサポートする必要がある場合があるため、これは重要です。

## セキュリティとレビューの要件

サンプリング仕様では人間によるレビューが重視されています。アプリケーションでは、サンプル リクエストのレビュー、送信前のリクエストの編集、配信前に生成されたレスポンスのレビューが簡単にできるようにする必要があります。

それは深刻な要求です。そうしないと、サンプリングが、ユーザーがほとんど理解できないサーバーによってトリガーされる隠れたモデル呼び出しになる可能性があります。|リスク |なぜそれが重要なのか |手すり |
|
---|
---|
---|
|隠れた急速噴射 |サーバーには安全でない命令が含まれている可能性があります。実行前に指示を表示 |
|データ漏洩 |コンテキストには機密情報が含まれる可能性があります。ユーザーが共有コンテキストを確認できるようにする |
|驚きのコスト |モデル呼び出しは有料トークンを消費する可能性があります。可視化モデルとコストポリシー |
|間違った出力 |生成された応答は不正確である可能性があります。最終納品前のレビュー |
|サーバーのオーバーリーチ |サーバーは不要なモデル作業を要求する可能性があります。明示的な機能サポートが必要 |

## AEO への影響

MCP サンプリングはクリーンなコンテキストに報酬を与えます。ドキュメント、API 応答、製品データ、またはワークフローの説明があいまいな場合、顧客モデルの仕事はさらに難しくなります。サンプリングベースのワークフローが構造化され、簡潔で、内部的にリンクされている場合、より優れた下流の応答を生み出すことができます。

Web サイト所有者にとって、これは次のことを意味します。

- 明確な定義を公開する
- API 応答の構造化を維持する
- 機械可読文書を公開する
- 散文の中に重要な制限を埋め込まないようにする
- 真実の文書ソースページ
- エンティティとアクションに安定した識別子を提供します

[AEO 開発者ガイド](/es/docs/developers-guide-aeo/) は当然の次のステップです。

## よくある質問

### MCP サンプリングを使用すると、サーバーは必要なモデルを呼び出すことができますか?

いいえ。クライアントはモデルへのアクセスを制御します。サーバーは設定を表現できますが、何が利用可能で何が許可されるかはクライアントが決定します。

### MCP サーバーにはサンプリングが必要ですか?

いいえ、クライアントの容量です。多くの MCP ワークフローでは、サンプリングなしでツール、リソース、プロンプトを使用できます。

### 本番環境でのサンプリングは安全ですか?

慎重に使用できますが、簡単なレビュー、コンテキストのレビュー、ログ、権限の制限、賢明なデフォルト設定が必要です。

### AEO にとってサンプリングが重要なのはなぜですか?

これは、エージェント ワークフローが、ページを取得するだけでなく、実行中にモデルにコンテンツについて推論するよう要求できることを示しています。構造化され、ソースに裏付けられたコンテンツがより有用になります。

## 結論MCP サンプリングは、ツールの実行とモデル推論の間の便利な橋渡しとなります。バックグラウンドでのショートカットではなく、制御された機能として扱います。
