# エージェント購入と最適化のための e コマース ガイド

顧客は AI アシスタントに「返品無料で 40 ユーロ未満の青い綿のシャツを買ってください」と言いました。エージェントは数秒で 50 店舗をスキャンします。製品データを確認し、在庫を確認し、返品ポリシーを読み、価格を比較して購入を完了します。プロセス全体にかかる時間は、人間がブラウザを開くよりも短くなります。

このセールで優勝する店は、最高の写真や最も知的な広告コピーを持った店ではありません。これは、すべてのデータ ポイントが明示的であり、すべてのポリシーが機械で読み取り可能であり、支払いプロセスがブラウザなしで機能する倉庫です。

これは[エージェント取引](/es/docs/agentic-commerce/)が実際にどのように見えるかです。これがオンライン ストアが変えなければならないことです。

## 製品データは完全で、構造化され、最新である必要があります。

エージェントは写真を見て素材を判断することはできません。寸法から重量を推測することはできません。すべての属性を構造化された方法で明示的に設定する必要があります。

各製品について、少なくとも以下を提示します。

- 名前、ブランド、カテゴリー
- 価格と通貨 (税金規定あり)
- 正確なサイズ、重量、素材、色 (商品名ではなく標準化された値を使用)
- バリアント ロジック (どのサイズが利用可能か、どの色が利用可能か)
- 発送時の寸法と重量
- 互換性の制限
- 状態（新品、再生品、中古品）

JSON-LD オファーと製品マークアップを使用します。しかし、さらに言えば、実際の在庫が 0 個であるにもかかわらず、構造化データが「在庫あり」と表示されている場合、エージェントの信頼は永久に失われることになります。データの完全性よりもデータの正確さが重要です。

属性の命名に一貫性がないことも同様に有害です。ある製品に「バッテリー寿命」と書かれ、別の製品に「実行時間」と書かれている場合、エージェントはそれらが同じことを意味しているかどうかを推測する必要があります。カタログ全体で語彙を標準化します。

## リアルタイムの在庫はオプションではありません

古い在庫シグナルは、エージェントの信頼を失う最も早い方法です。

人間は在庫切れのお詫びメールを受け入れるかもしれません。エージェントはそれをシステム障害として記録します。次回エージェントがオプションを比較するとき、あなたのストアは優先順位を失います。

専用のエンドポイントを通じて、または少なくとも数時間ではなく数分で更新される正確なスキーマ マークアップを通じてインベントリを公開します。正確な数が現実的でない場合は、在庫あり、在庫限り、予約注文、在庫切れなどの正直なステータス グループを使用します。

[E コマース向けの AEO 戦略トップ 10](/es/docs/aeo-ecommerce/) では、在庫の信頼性について詳しく説明しています。

## 自動トランザクションのためのヘッドレス チェックアウト従来のチェックアウト フローは、視覚的なステップ、進行状況バー、フォーム フィールド、確認モーダルなど人間向けに設計されています。エージェントにはそんなものは必要ありません。独自の文書化された API 呼び出しが必要です。

ヘッドレス支払いエンドポイントは以下を受け入れます。

- 製品識別子と数量
- 配送先住所
- 支払い認証情報 (生のカードデータではなく、安全なトークン経由)
- 配送方法の選択
- クーポンまたは割引コード (該当する場合)

注文 ID、配達予定日、請求総額、領収書の URL を含む確認を返します。

支払いが JavaScript、CAPTCHA、Cookie 同意モードでレンダリングされたブラウザ セッションを通じてのみ機能する場合、エージェントは購入を完了できません。彼らは、機械でチェックアウトできる競合他社から購入するでしょう。

## ポリシーは法的文書ではなく意思決定要素である必要があります返品ポリシー、配送保証、保証、およびキャンセル条件は、代理店向けの脚注ではありません。それらは選択基準です。

「30 日間返品無料の店舗からのみ購入する」ようにプログラムされたエージェントは、ミリ秒単位でその条件を確認する必要があります。法律文書の長い段落は役に立ちません。構造化ポリシー データは次のことを行います。

- 返品期間は数日です
- 返品費用（無料、有料、条件付き）
- 返金方法（当初の支払い、ストアクレジット）
- 保証期間
- 発送保証（X日以内の配達または返金）

構造化データとして埋め込むか、専用のポリシー エンドポイントを通じて公開します。ポリシーを機械で読み取り可能にしているブランドは、エージェント仲介のターゲティングにおいて目に見える利点を獲得します。 [実行層](/es/docs/execution-layer/) は、人間が判読できるものから機械が処理できるものへの移行が AEO の中核である理由を説明しています。

## 価格比較は決定的でなければなりません

エージェントは店舗間の価格を比較します。実際の数値を表示するために価格に JavaScript レンダリング、ログイン、または地理センシングが必要な場合、エージェントは間違った価格を抽出するか、価格をまったく抽出しない可能性があります。

実際の価格 (該当する場合は税金を含む) をページのソースと構造化データに表示します。段階的な価格設定や数量割引を提供する場合は、単なる視覚的な表ではなく、機械が読み取り可能な形式で価格設定ルールをレイアウトします。

## よくある質問

**代理購入とは何ですか?**
エージェント ショッピングとは、AI エージェントがユーザーに代わって製品を調査、比較、購入するプロセスを指します。エージェントは、製品の発見から支払い、注文の確認までのワークフロー全体を処理します。

**AI エージェントにとってヘッドレス決済が重要なのはなぜですか?**
AI エージェントは、ビジュアル チェックアウト フロー、JavaScript フォーム、または CAPTCHA と対話できません。ヘッドレス チェックアウト API を使用すると、エージェントは文書化された単一の API 呼び出しを通じて購入を完了できます。**返品ポリシーを機械で読み取り可能にするにはどうすればよいですか?**
保険契約条件を構造化されたデータ ポイントとして表現します (返品期間 (日数)、費用、返金方法、および条件)。解釈が必要な自然言語の段落に依存することは避けてください。

**在庫データが不正確な場合はどうなりますか?**
エージェントは不一致をシステム障害として記録します。ストアは信頼を失い、今後の比較において優先順位を失います。正確な在庫データは、単なる運用上の問題ではなく、最適化要素です。

**AI エージェントの購入には別の API が必要ですか?**
必ずしも分離している必要はありませんが、ショッピング フローはブラウザーに表示されずにアクセスできる必要があります。支払いに JavaScript、Cookie、または視覚的なインタラクションが必要な場合、エージェントには表示されません。
