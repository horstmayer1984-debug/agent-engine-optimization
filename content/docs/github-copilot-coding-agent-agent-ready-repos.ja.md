# GitHub Copilot コーディング エージェント: エージェント対応リポジトリを準備する方法

GitHub Copilot コーディング エージェントは、リポジトリの品質をエージェントのパフォーマンスの一部にします。問題が曖昧で、テストが不安定で、構成が文書化されていない場合、非同期コーディング エージェントは時間を無駄にしたり、危険な変更を加えたりすることになります。エージェント用のリポジトリを準備するということは、タスク、制限、コマンド、および検証手順を明示することを意味します。

## GitHub が発表した内容

GitHub は、Microsoft Build 2025 で [GitHub Copilot 用コーディング エージェント](https://github.com/newsroom/press-releases/coding-agent-for-github-copilot) を発表しました。この発表では、GitHub に組み込まれ、VS Code からアクセスできる非同期コーディング エージェントについて説明しており、作業は GitHub の開発層と制御層を通じて実行されます。

SEO/AEO の観点は実用的です。ソフトウェア システムは人間の開発者だけでなく、自律的な開発エージェントによっても読み取れる必要があります。

関連資料については、[開発者向けの AEO ユースケース](/es/docs/aeo-use-cases-developers/)、[エージェント対応 Web アプリケーション](/es/docs/agent-ready-web-apps/)、および [エージェント向けの MCP と API](/es/docs/mcp-vs-api-for-agents/) を参照してください。

## 「エージェント対応リポジトリ」とはどういう意味ですか?

エージェント対応リポジトリは、チームのルールを推測することなく、簡単に検査、変更、テストし、自律コーディング エージェントにレビューのために返すことができます。

含まれるもの:

- 明確なセットアップ手順
- 合格基準を備えた発行テンプレート
- 安定したIQチェック
- 文書化されたテストコマンド
- 簡潔なアーキテクチャに関するメモ
- 予測可能なファイル構成
- 安全な秘密の処理
- 生成された変更のルールを確認する

##エージェント対応の問題テンプレート

|問題フィールド |なぜ役立つのか |
|
---|
---|
|問題文 |広範囲の徘徊を防止 |
|範囲 |変更すべきものと変更すべきでないものを定義する |
|合格基準 |エージェントに終了ラインを与える |
|関連ファイル |探査コストを削減 |
|テストコマンド |検証を許可します |
|リスクに関する注意事項 |ブランドの移行、認証、支払いまたはデータ管理 |
|設計上の制約 | UI とコピーの一貫性を保つ |

問題は通知ですが、これはプロジェクト管理の産物でもあります。両方として扱います。

## リポジトリ準備チェックリスト

1. 「README」設定を最新の状態に保ちます。
2. 地域の慣習に対する「貢献」またはエージェントの指示を追加します。
3. パッケージ マネージャーとランタイムのバージョンを文書化します。
4. テスト コマンドを単独で安全に実行できるようにします。
5. CI が人間が期待するのと同じコマンドを実行することを確認します。
6. 古いスクリプトと非アクティブなドキュメントを削除します。
7. 意味のあるエラー メッセージを使用します。
8. 例に秘密を入れないでください。
9. フロントエンド作業のためのスクリーンショットまたはビジュアル要件を追加します。
10. 小さくてレビュー可能なタスクを使用します。

[AEO 開発者ガイド](/es/docs/developers-guide-aeo/) では、システムを一目瞭然にするという同じ原則が使用されています。

## 従来の開発者エクスペリエンスとの違い開発者エクスペリエンスは、人間の作業を高速化することに重点を置いています。エージェント対応の開発者の専門知識は、マシンが誤った想定を回避するのにも役立ちます。|クラシックDX |エージェントレディDX |
|
---|
---|
|人間はチームメイトに尋ねることができます。エージェントには書面によるコンテキストが必要です |
|非公式の構成知識が役に立ちます |構成は再現可能である必要があります。
|査読者は不足している証拠を後で発見する |エージェントには事前にテスト コマンドが必要です。
|高額な請求書も管理可能 |範囲が狭いチケットの方が効果的です。
|文書は物語的なものにすることができます |ドキュメントには実行可能ファイルの詳細が必要です。

これによって人間によるレビューが排除されるわけではありません。レビューがより集中的になります。

## 公開 Web サイトの AEO レッスン

同じ設計パターンがコードの外側にも適用されます。

- ページには明確な目的が必要です。
- アクションには目に見える制限が必要です。
- データは構造化されている必要があります。
- 成功状態は明示的である必要があります。
- 内部リンクは関係を示す必要があります。
- 機械可読なインデックスは重要なページを指す必要があります。

そのため、[llms.txt](/es/docs/llms-txt-vs-robots-txt/) と [実行層](/es/docs/execution-layer/) が重要です。エージェントには振動ではなく経路が必要です。

## よくあるエラー

次のパターンは避けてください。

- 一般的な「このアプリの改善」に関する問題を割り当てる
- 受け入れ基準がありません
- ドキュメントに記載されていないローカル ツールに依存する
- 無関係な理由で CI が失敗するようにする
- 古い紙幣のレイアウトルールを非表示にします
- セキュリティ保護を必要としないコードをセキュリティ障壁なしで編集するようエージェントに依頼します
- レビューなしでエージェントの出力をマージします

## よくある質問

### GitHub Copilot コーディング エージェントは大規模なチームにのみ有用ですか?

いいえ、小規模チームにもメリットはありますが、クリーンな問題と信頼性の高いテストが必要です。そうしないと、エージェントがレビュー負荷の別の原因になります。

### エージェントが対応できるドキュメントは人間によるレビューに代わるものですか?

いいえ。回避可能なエラーが減少します。アーキテクチャ、セキュリティ、製品評価、エッジケースについては、人間によるレビューが依然として必要です。

### エージェント対応リポジトリの最速のアップグレードは何ですか?

範囲、受け入れ基準、関連ファイル、テスト コマンドを含む明確な問題テンプレートを追加します。

### このトピックが AEO サイトに属するのはなぜですか?

エンコーディング エージェントは実行層の一例です。これらは、エージェントが指示、権限、検証ワークフローをどのように解釈するかを示しています。

## 結論

GitHub Copilot コーディング エージェントは、退屈なリポジトリの衛生管理に価値を追加します。システムが明確であればあるほど、システム内の自律エージェントはより適切に機能します。
