# ヘッドレス CMS を使用する場合

構造化コンテンツ、マルチチャネル パブリッシング、カスタム インターフェイス コントロール、または AI 読み取り可能なコンテンツ配信が明確なビジネス価値を生み出す場合は、ヘッドレス CMS を使用します。アーキテクチャの柔軟性よりも、シンプルな Web サイト、標準的な編集ワークフロー、メンテナンスの負担が少ないことが重要な場合は、これを避けてください。

## 決定ルール

柔軟性が追加のエンジニアリングに値する場合、ヘッドレスは良い選択肢です。チームが最新のサウンドの CMS を必要としているだけの場合、これは悪い選択です。

より広い定義については、[ヘッドレス ソフトウェアとは何ですか?](/es/docs/what-is-headless-software/) を参照してください。直接の比較については、[CMS とヘッドレス CMS](/es/docs/cms-vs-headless-cms/) を参照してください。

## これらが当てはまる場合はヘッドレス CMS を使用してください

|信号 |なぜそれが重要なのか |
|---|---|
|コンテンツは複数のチャネルにフィードされます |ヘッドレス コンテンツはエージェントの Web サイト、アプリ、ポータル、ドキュメントを強化できます |
|フロントエンドをカスタマイズする必要があります |開発者はテーマ システムに制限されません。
|コンテンツは構造化されています |フィールド、エンティティ、参照は再利用可能 |
|パフォーマンスは戦略的です |静的生成と CDN 配信が役立ちます |
| AI の最適化は重要 |構造化コンテンツは AI および AEO 検索をサポート可能 |
|開発者が利用可能です |統合、プレビュー、展開、QA には所有権が必要 |

##これらが当てはまる場合はヘッドレス CMS を避けてください

|信号 |最良の選択肢 |
|---|---|
|シンプルなウェブサイトで十分 |従来の CMS |
|編集者はページを視覚的に制御する必要があります |従来の CMS またはハイブリッド CMS |
|フロントエンド開発能力がない |従来の CMS |
|予算が厳しい |従来の CMS |
| SEOの基本はまだ解決されていない |まず現在のサイトを改善します |
|移行には URL 計画はありません |移行を遅らせる |

[AEO 実装ガイド](/es/docs/implement-aeo/) は、AI の準備が意思決定の一部である場合に役立ちます。

## SEO の準備に関する質問

ヘッドレスを選択する前に、次のことを確認してください。

1. タイトルとメタフィールドの所有者は誰ですか?
2. 正規版はどのように生成されますか?
3. ページはサーバー側でレンダリングされますか? それとも静的にレンダリングされますか?
4. 内部リンクはどのようにモデル化されますか?
5. XML サイトマップはどのように更新されますか?
6. リダイレクトはどのように管理されますか?
7. 画像の代替テキストはどのように保存されますか?
8. プレビューのインデックス作成をブロックするにはどうすればよいですか?
9. 構造化データはどのように生成されますか?
10. Search Console はリリース後にどのように監視されますか?

チームがこれらの質問に答えられない場合は、移行する準備ができていません。

## AI 最適化に関する質問

コンテンツ モデルに以下が含まれる場合、ヘッドレスは [AI 検索の最適化](/es/docs/ai-search-optimization/) に役立ちます。

- 定義
- よくある質問
- ソースリンク
- 比較基準
- エンティティ
- 人間関係
- 日付とバージョン
- 次のステップのアクション

ただし、公開ページは引き続き読み取れる必要があります。プライベート CMS API 内に適切に構造化されたコンテンツをトラップしないでください。## 決定例

|シナリオ |推薦 |
|---|---|
| 20 ページのローカル ビジネス サイト |従来の CMS |
|アプリ、Web サイト、サポート ボットの SaaS ドキュメント |ヘッドレスCMS |
|カスタム製品ストーリーテリングによる e コマース ブランディング |ヘッドレスまたはハイブリッド |
|技術者以外の編集者が運営するブログ |従来の CMS |
|多くの地域のエンタープライズ コンテンツ プラットフォーム |ヘッドレスまたはハイブリッド |
|ホーム ホームページ |従来の CMS または静的サイト |

## よくある質問### ヘッドレス CMS を使用する必要があるのはどのような場合ですか?

構造化コンテンツ、マルチチャネル配信、カスタム インターフェイスのニーズ、または AI 読み取り可能なパブリッシングにより、さらなる複雑さが正当化される場合に使用します。

### ヘッドレス CMS を避けるべきなのはどのような場合ですか?

迅速なセットアップ、簡単な編集、低コスト、標準的な Web サイトが主な目標である場合は、これを避けてください。

### ヘッドレス CMS は SEO に適していますか?

はい、SEO がコンテンツ モデルとインターフェイスに統合されている場合は可能です。メタデータ、レンダリング、リンク、サイトマップが二次的なものとして扱われる場合はそうではありません。

### ヘッドレス CMS は AI の最適化に適していますか?

そうかもしれません。構造化コンテンツは AI システムに役立ちますが、それは最終サイトが明確でクロール可能な、ソースに裏付けられたページを公開している場合に限られます。
