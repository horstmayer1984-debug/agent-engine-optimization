# 金融サービス向け AEO: 代理店の準備

金融サービス AEO とは、AI エージェントが責任を持って比較し指示できるよう、製品、開示、適格性ルール、サポート パス、許可されるアクションを明確にすることを意味します。目標は、エージェントがレビューされていない財務上のアドバイスを提供できないようにすることです。目標は、規制対象の決定を適切な管理下に保ちながら、製品、制限、料金、次のステップに関する公開データを機械で読み取り可能にすることです。

## 金融業界にはより厳格な AEO モデルが必要な理由

金融サイトでは、人間にとっては便利に見えても、エージェントにとっては曖昧な商品ページが公開されることがよくあります。料金は脚注に記載されており、資格は PDF に分割され、料金ページは製品ページとは別に更新され、開示情報を関連するオファーに結び付けるのは困難です。

アカウント、カード、ローン、またはアドバイス サービスを比較するエージェントには、決定的なフィールドが必要です。

- 製品タイプ
- 資格要件
- 料金と料金条件
- 管轄区域または居住地の制限
- 必要書類
- リスクの開示
・応募方法
- 人間の登山ルート

[実行層ガイド](/es/docs/execution-layer/) では、エージェントが説得力のあるコピーだけではなく、現在の構造化データを必要とする理由を説明しています。

## 金融 AEO の使用例

|使用例 |エージェントに必要なもの |より安全な最初のアクション |
|
---|
---|
---|
|銀行口座を比較 |手数料、限度額、入金ルール、アカウントの機能 |製品ページを推奨する |
|マッチングローンオプション |資格、年利範囲、期限、必要書類 |承認ではなくルートの事前検証 |
|アドバイザリーサービスを探す |資格情報、サービス モデル、最小要件、場所 |相談のスケジュール |
|既存の顧客をサポート |認証されたアカウントのコンテキスト |安全なポータルへのルート |
| FinTech API の比較 |ドキュメント、価格設定、稼働時間、認証モデル |統合設定を返す |

最も安全な AEO パターンは、公開比較と制御された実行です。エージェントは、製品に関する公開データを読んで比較できます。口座開設、取引、与信判断、顧客サービスなどの機密性の高いアクションは、人間が承認した記録とルールを備えた認証システムに移行する必要があります。

## レイヤ要件を読む

金融商品ページは単なるマーケティング ページではなく、データ シートとして構造化する必要があります。

各ページには以下を含める必要があります。

・わかりやすい商品名
- 製品カテゴリー
- 誰のための製品ですか?
- 料金と条件
- 資格規定
- 国または州ごとの入手可能性
- 必要な開示
- 最終更新日
- 連絡先またはリクエストルート
- 平易な言葉でのリスクメモ

AI 検索および応答システムの場合は、簡潔なタイトル、FAQ ブロック、および関連製品への内部リンクを使用します。 [AEO、SEO、GEO の比較](/es/docs/aeo-vs-seo-vs-geo/) は、財務チームが分類、AI 引用、機械動作の準備という 3 つのレイヤーすべてを必要とすることが多いため、ここで役立ちます。

## 実行層の要件

組織が承認、登録、レビューのモデルを確立するまでは、リスクの高い金融行為を公開しないでください。|容量 |初期の AEO に適しています |より強力なガバナンスが必要 |
|
---|
---|
---|
|製品検出 API |はい |公開されている場合はリスクが低い |
|料金と料金 API |はい |更新してバージョン管理する必要があります。
|事前の資格確認 |たぶん |それを承認要求に変えることは避けてください |
|アプリケーション ホーム |たぶん |身元確認と同意チェックが必要 |
|アカウントアクション |一般公開禁止 |認証と監査証跡が必要 |
|投資または融資の決定 |一般公開禁止 |規制された意思決定ガバナンスが必要 |

エージェントには明示的な容量制限を与える必要があります。例: 「このエンドポイントは現在の製品条件を返すことができます。クレジットを承認したり、口座を開設したり、個別の財務アドバイスを提供したりすることはできません。」

## 信頼とガバナンスの兆候

金融 AEO は、量よりも信頼性に大きく依存します。代理店は、オファーが最新であること、機関が特定可能であること、説明する製品に必要な開示が添付されていることを確認できなければなりません。

役立つ外部参照:

- [SEC: 規制 S-P ルール作成ページ](https://www.sec.gov/rules-regulations/2024/06/s7-05-23)
- [SEC: 規則 S-P の修正の発表](https://www.sec.gov/newsroom/press-releases/2024-58)
- [NIST AI リスク管理フレームワーク](https://www.nist.gov/itl/ai-risk-management-framework)

SEC 規則 S-P 資料は、顧客情報の保護とプライバシー要件に対応しているため、SEC 規則の対象となる金融機関に関連します。これらはすべての企業に共通のルールではないため、各組織は依然として独自の法的レビューを必要としています。

## 導入チェックリスト

1. 現在のレート、料金、資格ルールを記載した構造化された製品ページを作成します。
2. 必要に応じてアウトラインを追加し、重要な情報をクロール可能な HTML に保持します。
3. 製品カテゴリ、サポート パス、およびエージェントの制限を含む「llms.txt」を公開または更新します。
4. パブリック比較エンドポイントを認証されたクライアント アクションから分離します。
5. バージョン管理を追加し、料金、開示、API 応答にタイムスタンプを更新します。
6. ワークフローをトリガーするエージェントのリクエストを記録します。
7. コンプライアンス、セキュリティ、および製品所有者とともに各アクションを確認します。

[AEO 準備状況チェッカー](/tools/aeo-readiness-checker.html) は、実行層でのより深い作業が開始される前に、パブリック検出層を評価できます。

スコープに認証されたアクション、ポリシー制御、または規制された意思決定パスが含まれる場合は、[完全な AEO 準備監査](/es/docs/audit/) を使用します。

## よくある質問

### AI エージェントは金融商品を推奨できますか?公開情報を比較することはできますが、個別の財務上の推奨事項により、規制、適合性、または受託者に対する懸念が生じる可能性があります。 AEO ページでは、一般情報とアドバイスを明確に区別する必要があります。

### 銀行またはフィンテックにとって最初の AEO プロジェクトは何ですか?

製品に関する構造化データと情報から始めます。これにより、機密性の高いアカウントのアクションを公開することなく、AI の可視性が向上します。

### 金融サービス会社はエージェント向けの API を公開する必要がありますか?

製品、レート、サポート ルーティングのパブリック API を公開できます。アカウント固有のアクションは、厳格な制御を備えた認証されたシステムの背後に残す必要があります。

### AEO はフィンテック SEO とどう違うのですか?Fintech SEO はランキングとトラフィックを最適化します。 AEO は、AI システムが製品ルールを理解し、オプションを比較し、ユーザーを次の正しいステップに安全に誘導できるかどうかを最適化します。

### エージェントに絶対に隠してはいけないものは何ですか?

料金、資格制限、リスク開示、管轄区域の制限、更新日。エージェントがそれらを見ることができなければ、責任を持って製品を比較することができません。
