# 比較したエージェント支払いプロトコル: x402、ACP、MPP、AP2、TAP、エージェント支払い

エージェントの支払いプロトコルは、2026 年に決済、取引オーケストレーション、信頼の 3 つのレイヤーに分割されます。 x402 は、軽い HTTP 決済やステーブルコイン決済ではより強力です。 ACP と UCP はショッピング ジャーニーをカバーします。 MPP は自動継続支払いを目指しています。 AP2、Visa TAP、Mastercard Agent Pay は、カード ネットワークにおける認証、アイデンティティ、信頼に重点を置いています。

支払いはもはや単なる支払い詳細ではないため、これはエージェント エンジンの最適化にとって重要です。 AI エージェントがオファーを発見できても、支払い、承認、トランザクションの確認ができない場合、サイトは表示されますが、実行可能ではありません。

## 2026 年 5 月に何が変わったのでしょうか?

市場にはもはや明確な支払い基準がありません。バッテリーが付いています。

Linux Foundation が Coinbase がプロトコルに貢献する x402 Foundation を発表した後、x402 はネイティブ暗号実験からより広範なインフラストラクチャに移行しました。 Stripe は自動支払いに関するドキュメントを公開し、Tempo による MPP を導入しました。 OpenAI と Stripe により、ACP がインスタント チェックアウトの背後にあるビジネス層になりました。 Google は AP2 を認証プロトコルとして推進し、その後 FIDO Alliance を通じて AP2 を標準ガバナンスに移行しました。 Visa と Mastercard は、カード ネットワークの信頼モデルを代理店支払いの会話に取り入れました。実際的な結論は単純です。販売者は「どのプロトコルが勝つのか?」と尋ねるべきではありません。 「このプロトコルはどのレイヤーを解決しますか?」と尋ねる必要があります。

## 集落層: x402

x402 では、エージェントと API の支払いに HTTP 402 Payment Required パターンが復活します。クライアントは保護されたリソースを要求し、サーバーは支払い要件を返し、クライアントは支払い証明を添付し、サーバーは検証後にアクセスを許可します。

x402 は、製品が小型、デジタル、API ベース、またはリクエストごとに価格設定されている場合に強力です。天気エンドポイント、プレミアム データ ソース、MCP ツール、推論呼び出し、または機械可読レポートは、アカウントを作成しなくても呼び出しごとに料金を請求できます。その限界も明らかです。 x402で値が確定します。購入プロセス全体を単独で解決できるわけではありません。販売者は依然として、製品の発見、ポリシーの開示、不正行為の管理、返金、身元確認、税金、サポート、紛争管理を必要としています。

サイト ガイドをお読みください: [Agent Payments x402](/es/docs/x402-agent-payments/)。

## コマース層: ACP および UCP

ACP と UCP は、x402 よりも購入プロセスに近いものです。

OpenAI と Stripe によって開発された ACP は、エージェントが開始する支払いを中心に設計されています。 Stripe は、エージェント コマースについて、フロー内で共有の支払いトークンとカタログ管理を使用して、バイヤー、エージェント、企業が安全に取引できる方法であると説明しています。

UCP はさらに広範囲です。これは、ディスカバリー、カタログのやり取り、支払い、購入後のサポートを通じたエージェント購入のための共通のビジネス言語として位置付けられています。これにより、支払い方法というよりは、商用プロトコルの表面に近いものになります。

e コマース チームにとって、問題が有料 API リクエストではなく通常の小売購入である場合、x402 よりも ACP と UCP の方が関連性が高くなります。

## マシン決済レイヤー: MPP

Stripe と Tempo が共同執筆した MPP は、人間に知られずに企業やその他のエージェントに支払いを行うエージェントを対象としています。その最も強力な使用例は、1 回限りの支払いではありません。これは、反復的、継続的、または測定されたアクティビティです。

それは自律システムに適しています。エージェントは、今後のコールの正確な数を常に知っているわけではありません。見積もり、期間、料金上限、およびジョブ完了後の調整が必要になる場合があります。

x402 が 1 つのリクエストに対してクリーンであれば、MPP は一連のリクエストに対して興味深いものになります。

## 認可レイヤー: AP2

AP2 は、信頼の問題に対する Google の答えです。販売者は、ユーザーが代理店にこのような条件で商品を購入することを許可したことをどのようにして知るのでしょうか? Google は、AP2 をプラットフォーム間でエージェント主導の支払いを安全に開始および処理するためのオープン プロトコルであると説明しています。その中心的な概念は、ユーザーの意図、カートの詳細、および支払いの承認を捕捉する、暗号的に検証可能な証拠である義務です。

AP2 は支払い方法とはあまり関係がなく、責任と関係があります。カード、ウォレット、暗号通貨決済レールと連携できます。そのため、純粋な代替品ではなく、x402 を補完するものになります。

## カード ネットワーク層: Visa TAP および Mastercard Agent Pay

Visa TAP と Mastercard Agent Pay は、既存のカード ネットワークがエージェント主導の商取引にどのように適応するかという別の問題を解決します。

Visa TAP は、正規の AI エージェントを不審な自動トラフィックから区別し、取引フロー全体で資格情報とエージェントのコンテキストを渡す方法を販売者に提供します。 Mastercard Agent Pay は、カードコマースですでに使用されているトークン化の概念を取り入れ、それをエージェントの支払いに適用します。

多くのトレーダーはステーブルコインから始めないため、これらのシステムは重要です。彼らは、すでに信頼している既存のカード、プロセッサ、トークン化された認証情報、および詐欺システムから始めます。

## プラットフォーム層: 採掘されない

Nevermined は単なるプロトコルではありません。これは、エージェント API、MCP ツール、保護された資産への有料アクセスのためのインフラストラクチャです。彼らのドキュメントでは、x402 フロー、Stripe 支払い、ステーブルコイン、プラン、資格チェックのサポートについて説明しています。

そのため、Nevermined は、すべてのコンポーネントを自分たちで構築する必要がなく、便利なエージェント支払いレイヤーを必要とするチームに適しています。## 企業はどのプロトコルを選択すべきでしょうか?

## AEO への影響

AEO の勝ちパターンはハイブリッド スタックです。

サイトは、機械可読な製品またはサービス データを公開し、決定的なアクション パス、承認および支払い要件を文書化して、結果を検証可能にする必要があります。支払いプロトコルは、エージェントがサイトが何を提供しているのか、どのような行動をとれば安全なのかを理解した後にのみ役に立ちます。次の内部ステップ: [AI エージェント プロトコル](/es/docs/protocols/)、[実行層](/es/docs/execution-layer/)、および [x402 MCP A2A フル スタック](/es/docs/x402-mcp-a2a-full-stack/) を読みます。

## よくある質問

**2026 年に最適なエージェント支払いプロトコルは何ですか?**
単一の最適なプロトコルはありません。 x402 は軽量の HTTP 決済に対してより強力であり、ACP と UCP はビジネス フローに対してより強力であり、MPP は継続的な自動支払いに対応し、AP2 は承認と信頼に重点を置いています。

**x402 は ACP よりも優れていますか?**
x402 と ACP は異なるレイヤーを解決します。 x402 は HTTP 経由での支払い決済を処理します。 ACP は、エージェントが開始する支払いと販売者の統合を処理します。

**AP2 は支払い方法ですか?**
AP2 は、認可および信頼プロトコルとして最もよく理解されています。さまざまな支払い方法と決済方法に対応できます。

**Visa と Mastercard は x402 と競合しますか?**
これらはいくつかのユースケースで競合しますが、さまざまな導入問題も解決します。カード ネットワークは、既存の加盟店が最初に暗号化ネイティブ レールに移行することなく代理支払いをサポートするのに役立ちます。

**販売者は最初に何を実装する必要がありますか?**
ほとんどの販売者は、構造化された商品データ、明確なポリシー、エージェントが読み取り可能な支払い要件から始める必要があります。プロトコルの選択はビジネス モデルに従う必要があります。
