llms.txt は、AI エージェントに Web サイトを検出させるための最も簡単かつ効果的な方法です。サイトのルートにあるマークダウン ファイルは、エージェントに、何を提供するか、どこで見つけられるか、どのように操作するかを伝えます。エージェントはサイト全体をクロールする前に llms.txt を検索するため、正しく処理するということは、最初に発見されることを意味します。

## 2026 年に llms.txt が交渉不可能になる理由

AI エージェントは、新しいサイトを見つけるときに検出階層に従います。まず、llms.txt を検索します。次に、agent-card.json を探します。 3 番目に、HTML 内の構造化データを探します。 4 番目に、生のページ コンテンツを解析してみます。

llms.txt のないサイトでは、エージェントはステップ 3 または 4 から開始することになりますが、これは時間がかかり、信頼性が低く、不正確な抽出が行われる可能性が高くなります。明確な llms.txt が含まれるサイトでは、エージェントは 1 回のリクエストで貴社のサービスの全範囲を理解できます。

Stripe、Cloudflare、および Cursor はすでに llms.txt ファイルを公開しています。早期導入者は、エージェントのエンゲージメント率が大幅に高く、AI が生成した応答の見積もりがより正確であると報告しています。

## 正確な構造

llms.txt はマークダウン ファイルです。情報を整理するには見出しを使用し、説明にはプレーン テキストを使用します。推奨される構造は次のとおりです。

**行 1: H1 ヘッダーとしてのサイト名。** 1 行、プレーン テキスト。

**セクション 1: このサイトが提供するもの** 製品、サービス、コンテンツを客観的な言葉で説明する短い段落 (2 ～ 3 文)。マーケティング言語はありません。それが何をするのか、誰のためのものなのか、そして何が役立つのかを説明します。

**セクション 2: 主要なページ** 最も重要な URL のリストと、それぞれについての 1 行の説明。ホームページ、主要な製品またはサービスのページ、価格設定ページ、ドキュメント、およびエージェントが特定の情報を探す必要があるページを含めます。

**セクション 3: 利用可能なエンドポイント** サイトで API エンドポイントまたは MCP ツールを公開している場合は、それぞれが何を行うか、必要な入力、および何を返すかについての簡単な説明を付けてリストします。

**セクション 4: 認証。** エージェントを認証する方法を示すか、読み取りアクセスには認証が必要ないことを明示します。

**セクション 5: 制限。** レート制限、使用ポリシー、データ更新の保証、およびエージェントがサイトとやり取りする方法に関する制限。

**セクション 6: 連絡先。** エラーを報告する方法、または説明をリクエストする方法。

## コンテンツのルール

説得するためではなく、抽出するために書きます。各文には、エージェントが使用できるファクトが含まれている必要があります。情報を伝えない形容詞は削除します。 「当社の有能なプラットフォーム」を「医療提供者向けの API ベースのプログラミング プラットフォーム」に置き換えます。説明は 1 つの項目につき 1 ～ 2 文にとどめてください。エージェントには段落は必要ありません。ユーザーのクエリと比較できる正確なステートメントが必要です。

重要な機能を追加、削除、または変更するたびに、llms.txt を更新します。提供しなくなった機能について説明した古い llms.txt は、エージェントが失敗するアクションを試行する原因となるため、llms.txt がないことよりも悪いです。

## 大規模サイト向けの動的生成

数百もの製品を扱うサイト、または在庫が頻繁に変更されるサイトでは、llms.txt を手動で管理するのではなく、動的に生成する必要があります。製品データベースにクエリを実行し、現在のカタログ概要を抽出し、ルート設定からアクティブな API エンドポイントを一覧表示し、結果をマークダウン ファイルとして書き込むスクリプトを作成します。このスクリプトはスケジュールに従って実行します (ほとんどのサイトでは毎日、在庫が多いサイトでは 1 時間ごと)。

Hugo ベースのサイトの場合、コンテンツ データをマークダウンとして表すカスタム出力形式を作成することにより、ビルド プロセスの一部として llms.txt を生成できます。

## llms.txt とスキーマ マークアップの統合

llms.txt とスキーマ マークアップは補完的な目的を果たします。 llms.txt は、サイトレベルの検出 (このサイト全体が提供するもの) を提供します。スキーマ マークアップは、ページ レベルの詳細 (この特定のページに含まれるもの) を提供します。

潜在的なアクション ターゲットとして Web サイトのスキーマに含めることにより、スキーマ マークアップから llms.txt を参照します。 llms.txt のキー ページ セクションで、スキーマでマークされた最も重要なページを参照します。

これにより、2 層の検出システムが作成されます。llms.txt で始まるエージェントはスキーママーク付きのページをより迅速に見つけ、特定のページで始まるエージェントは llms.txt に移動してサイトの全体像を表示できます。

## よくあるエラー

llms.txt をマーケティング テキストとして記述します。エージェントは説得に応じません。彼らは構造化された事実に反応します。各行は、サイトで何ができるかについての具体的な質問に答える必要があります。

サイト上のすべてのページのリスト。 llms.txt はサイトマップではありません。エージェントとの対話にとって最も重要なページのみを含めます。 500 ページのサイトの場合、llms.txt は 10 ～ 30 の主要なページを参照する必要があります。

更新するのを忘れてしまう。古いエンドポイントや製造中止になった製品を参照する llms.txt は、エージェントの信頼を著しく傷つけます。 [AEO 導入ガイド] (/docs/implement-aeo/) では、llms.txt がより広範な最適化パスにどのように適合するかを説明しています。

## よくある質問

**llms.txt は正確にどこに配置する必要がありますか?**
ドメインのルート: yourdomain.com/llms.txt。エージェントは、robots.txt と同様に、この正確なパスでファイルを探します。

**llms.txt にはどの形式を使用すればよいですか?**
割引。セクションには見出しを使用し、説明にはプレーンテキストを使用します。 HTML、JSON、カスタム形式はありません。

**llms.txt の長さはどれくらいですか?**
1 ページまたは 2 ページの簡潔なテキスト。主要なサービスとエンドポイントを説明するのに十分な時間。エージェントが 1 回の読み取りで処理できる程度に短い。

**llms.txt には価格情報を含めるべきですか?**
はい、価格が公開されている場合は可能です。基本価格、価格モデル (シートごと、使用ごと、定額料金)、および詳細な価格情報の入手先を含めます。

**サブドメインごとに異なる llms.txt を使用できますか?**
はい。各サブドメインには、そのサブドメインの特定の機能を説明する独自の llms.txt が必要です。
