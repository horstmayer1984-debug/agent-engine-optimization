開発ツールはすでに、構造化された、主にテキストベースの環境で動作します。ドキュメント、API 仕様、コード リポジトリ、および構成ファイルは、本質的にほとんどの Web コンテンツよりも機械可読です。これで AEO は有利にスタートできますが、それで仕事が完了したわけではありません。

ほとんどのドキュメントは、コンテキスト内で閲覧、検索、読み取りを行う人間向けに書かれています。 AI エンコード エージェントの相互作用は異なります。特定の情報をすばやく見つけて正確に抽出し、タスクに適用して結果を検証する必要があります。ドキュメントがそのワークフローをサポートすると、エージェントの効率が大幅に向上します。

## ドキュメントのナビゲーションと抽出。

「決済 API と Webhook サポートの統合」を任務とするコーディング エージェントは、関連するエンドポイント ドキュメントを検索し、必要なパラメーターを抽出し、認証要件を特定し、エラー処理を理解し、コード サンプルを見つける必要があります。

これは、ドキュメントが一貫した構造 (明確なメソッド/パス/パラメータ定義を含むエンドポイント参照、認証セクション、エラー コード リファレンス、および実際の例) に従っている場合にうまく機能します。重要な詳細が明確な相互参照なしにブログ投稿、変更履歴、FAQ ページに分散されている場合、問題は発生します。

AEO に最適化された開発者ドキュメントは、一貫した書式設定と明示的な相互参照を備えた、単一の構造化された信頼できる情報源を維持します。

## 仕様からのコード生成。

エージェントは API 仕様に基づいてコードを生成します。 OpenAPI (Swagger) 仕様、GraphQL スキーマ、およびプロトコル バッファー定義は、コード生成ワークフローへの入力として機能します。

生成されたコードの品質は、仕様の品質に直接依存します。不完全な仕様は不完全なコードを生成します。フィールドの説明があいまいだと、不正確な実装が発生します。欠落した例は幻覚パターンを生み出します。

開発ツールの AEO とは、API 仕様を完全で正確でバージョン管理され、維持されている第一級のドキュメントとして扱うことを意味します。

## ドキュメントとコードのデバッグ

開発者がバグを見つけると、AI エージェントがドキュメント、スタック オーバーフロー、GitHub の問題、およびコード ベースを検索して関連するコンテキストを探します。エージェントはバグを既知の問題と比較し、解決策を見つけて、それが現在のバージョンと構成に適用されることを確認する必要があります。

このワークフローは、構造化されたエラーのドキュメントに依存しています。各エラー コードには、原因、解決手順、影響を受けるバージョン、および関連する問題を示す特定のエントリが必要です。複数のエラーを 1 ページにまとめた構造化されていないトラブルシューティング ガイドは、エージェントの精度を低下させます。## マルチエージェント開発オーケストレーション

複雑な開発タスクには、複数のエージェントの協力が必要です。 1 人のエージェントがアーキテクチャを計画し、別のエージェントが実装を作成し、3 番目のエージェントがテストを生成し、4 番目のエージェントがセキュリティ問題をレビューします。

このチェーン内の各エージェントは、ドキュメントとコードベースのさまざまな部分にアクセスする必要があります。マルチエージェント開発の AEO は、プランナーのためのアーキテクチャ上の決定、実装者のための API 詳細、テスト作成者のためのテスト パターン、レビュー担当者のためのセキュリティ ガイドラインなど、モジュール化された役割に適した情報を公開することを意味します。記事 [マルチエージェント AEO](/es/docs/multi-agent-aeo/) では、オーケストレーションの最適化について詳しく説明しています。

## ツールと API の検出

エージェントは、入手可能な情報に基づいて、使用するツールとライブラリを選択します。明確なインストール手順、クイック スタート ガイド、構造化された機能の説明を備えた十分に文書化されたライブラリが、同等の機能はあるが文書化が不十分な代替ライブラリよりも選択されます。

これは、電子商取引における製品発見に相当する開発者です。 [AEO とは](/es/docs/what-is-aeo/) では、両方のドメインに適用される構造モデルについて説明します。

## よくある質問

**AEO は開発者ドキュメントにどのように適用されますか?**
AEO は、AI コーディング エージェントが情報を正確に特定、抽出、適用できるようにドキュメントを構造化します。一貫したフォーマット、明示的な相互参照、完全な API 仕様が重要です。

**API ドキュメント エージェントを準備するには何が必要ですか?**
完全な OpenAPI または同等の仕様、明確なパラメーターの説明、認証ドキュメント、解決策を含むエラー コード リファレンス、および実際に動作するコード サンプル。

**コーディング エージェントはデバッグをどのように処理しますか?**
エージェントはエラー メッセージを文書化されたエラー コードと比較し、解決手順を見つけて、現在のバージョンへの適用性を確認します。間違いを構造化した文書化により精度が向上します。

**AEO はエージェントが推奨するライブラリやツールに影響しますか?**
はい。エージェントはツールを選択する際に、ドキュメントの品質、完全性、構造を評価します。十分に文書化されたライブラリが最も頻繁に選択され、推奨されます。

**マルチエージェント開発は AEO とどのような関係がありますか?**
複数の専門エージェント (プランナー、コーダー、テスター、レビュー担当者) は、ドキュメントの異なるビューを必要とします。 AEO により、各役割が関連する構造化された情報に効率的にアクセスできるようになります。
