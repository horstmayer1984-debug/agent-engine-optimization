# ヘッドレス CMS と従来の CMS の比較

従来の CMS は、コンテンツ管理とページ レイアウトを 1 つのシステムに組み合わせています。ヘッドレス CMS は構造化コンテンツを保存し、API を介して別のインターフェイスに配信します。実際の違いは単純です。通常、パブリッシャーにとっては従来の CMS プラットフォームの方が使いやすいのに対し、ヘッドレス CMS プラットフォームは開発者、マルチチャネル パブリッシング、エージェントが読み取り可能なコンテンツ配信にとってより柔軟です。

## 従来の CMS とは何ですか?

従来の CMS は、編集者にページの作成、プレビュー、デザイン、公開を行う場所を提供します。 WordPress がその明らかな例ですが、このパターンは多くのホスト型 Web サイト ビルダーや従来のエンタープライズ CMS プラットフォームにも見られます。

強みは操作の簡単さです。編集者は、フロントエンド リポジトリを開かずに、ページの作成、ナビゲーションの更新、画像のアップロード、テンプレートの変更、公開を行うことができます。多くの Web サイトにとって、これは正しいトレードオフです。

従来の CMS プラットフォームは以下に適合します。

- 地元企業のウェブサイト
- ブログ
- サービスページ
- 中小企業のウェブサイト
- シンプルなランディングページ
- 標準的なレイアウトの小さな店舗

## ヘッドレス CMS とは何ですか?

ヘッドレス CMS は、最終プレゼンテーション層とは別にコンテンツを保存します。 CMS は、フィールド、エントリ、アセット、ワークフロー、およびロールを管理します。インターフェイスは API 経由でそのコンテンツを抽出し、それを Web サイト、アプリ、ポータル、またはその他のエクスペリエンスとして表示します。

Contentful は、このパターンを、コンテンツを 1 か所で管理し、任意のデジタル チャネルに配信できるように、プレゼンテーション層をバックエンドから分離すると説明しています。

このため、ヘッドレス CMS アーキテクチャは構造化コンテンツに特に役立ちます。製品定義、著者略歴、ヘルプ記事、またはコンプライアンスに関するメモは、システム間でコピーすることなく再利用できます。

より広い概念については、[ヘッドレス ソフトウェアとは?](/es/docs/what-is-headless-software/) を参照してください。

## SEO に最適な CMS の種類は何ですか?

どちらのモデルもデフォルトでは SEO に有利ではありません。検索のパフォーマンスは実装の品質に依存します。

有用なコンテンツは、実際の質問に答え、トピックを適切にカバーし、読者に合わせて構成され、独自性を維持する必要があります。これは、サイトが WordPress、Contentful、Sanity、Adobe Experience Manager、またはカスタム スタックで実行されているかどうかに当てはまります。

| SEO 要素 |従来の CMS |ヘッドレスCMS |
|
---|
---|
---|
|タイトルとメタディスクリプション |通常、プラグインまたは組み込みフィールドを通じて利用可能です。フロントエンドによってモデル化およびレンダリングされる必要があります。
|クロール可能な HTML |通常、デフォルトで存在します。レンダリング戦略に依存します |
|内部リンク |編集者にとっては簡単ですが、複雑になる可能性があります | || モデルとコンテンツ コンポーネントに組み込まれるように設計する必要があります ||パフォーマンス |適切なホスティングとキャッシュを使用すると高速になります |非常に高速ですが、自動的には実行されません。
|構造化データ |多くの場合、プラグインによって強化されます。全体的にパーソナライズされ、より制御可能 |
| SEO リスク |肥大化したプラグイン、弱いテーマ、重複ページ |メタデータの欠落、クライアント側のレンダリング、貧弱なプレビュー |

ヘッドレス CMS は、フロントエンドがサーバー側レンダリングまたは静的生成を使用し、クリーンな内部リンクを持ち、メタデータを公開し、JavaScript の背後に重要なコンテンツを隠すことを回避する場合、SEO に最適です。開発者がコンテンツをクロール可能なドキュメントではなくアプリケーション ペイロードとして扱うと、SEO にとって悪影響を及ぼす可能性があります。## 編集経験

従来の CMS プラットフォームは通常、すぐに使用できるスムーズなエクスペリエンスをサイト運営者に提供します。単一のワークフローでページ、ブロック、メニュー、プレビュー、およびドラフト状態を表示できます。

ヘッドレス CMS プラットフォームも堅牢な編集ワークフローをサポートできますが、ページのプレビューとレイアウトには計画が必要です。マーケティング部門が開発者の関与なしに毎日公開する必要がある場合、この点はアーキテクチャの純度よりも重要です。

## 開発者の経験

開発者は、インターフェイス フレームワークを選択し、再利用可能なコンポーネントを定義し、テーマ システムとの競合を避けることができるため、ヘッドレス CMS プラットフォームを好むことがよくあります。ヘッドレス コンテンツは、コンテンツを単にレンダリングされたページではなく構造化データとして公開できるため、[エージェント対応 Web アプリケーション](/es/docs/agent-ready-web-apps/) にも適しています。

トレードオフはメンテナンスです。ヘッドレス サイトには通常、個別のホスティング、展開、プレビュー ロジック、コンテンツ モデル、API クライアント、および監視が必要です。

## 従来の CMS を選択する必要があるのはどのような場合ですか?

マルチチャネルの柔軟性よりも速度、予算、サイト運営者の独立性が重要な場合は、従来の CMS を選択してください。これは通常、小規模な Web サイト、開発者のいないコンテンツ チーム、および複数のインターフェイスを必要としない組織に適した選択です。

## ヘッドレス CMS を選択する必要があるのはどのような場合ですか?コンテンツを複数のチャネルに提供する必要がある場合、インターフェイスが高度にカスタマイズされている場合、または組織がコンテンツを構造化データとして機能させたい場合は、ヘッドレス CMS を選択してください。これは、ドキュメント センター、SaaS サイト、国際的なコンテンツ プラットフォーム、顧客ポータル、AI 読み取り可能な出版システムに最適です。

関連ガイド [ヘッドレス コマースの説明](/es/docs/headless-commerce-explained/) では、オンライン ストアの同じバックエンド/フロントエンドの分割について説明しています。

## よくあるエラー

最大の間違いは、より先進的に聞こえるという理由でヘッドレスを選択することです。 2 番目に大きな間違いは、エディターのことを忘れてしまうことです。

ヘッドレスに移行する前に、以下を定義します。

- 再利用可能なコンテンツタイプとフィールド
- ページのプレビュー
- SEO メタデータ フィールド
- 内部リンクのルール
- 正規 URL ロジック
- スキーマのマークアップ
- 公開権限
- API が失敗した場合のフォールバック動作

[AEO 実装](/es/docs/implement-aeo/) では、エージェントがコンテンツを検出する方法と、Markdown、JSON、または API サーフェスが HTML ページとともに存在するかどうかも定義します。

## よくある質問

### ヘッドレス CMS は WordPress より高速ですか?

そうなる可能性はありますが、常にそうとは限りません。速度は、フロントエンド コード、レンダリング、画像、ホスティング、キャッシュ、サードパーティのスクリプトによって異なります。

### ヘッドレス CMS は大規模な Web サイトに適していますか?多くの場合、特にコンテンツがさまざまな国、アプリケーション、Web サイト、ポータル、またはドキュメント システム間で再利用される場合にはそうです。

### WordPress はヘッドレス CMS として使用できますか?

はい。 WordPress は API を通じてコン​​テンツを公開できますが、その設定では WordPress の魅力であるシンプルさが一部失われます。

### ヘッドレス CMS は初心者に適していますか?

一般的にはそうではありません。セットアップ、プレビュー、メンテナンスが簡単なため、初心者は従来の CMS からより多くの価値を得ることがよくあります。
