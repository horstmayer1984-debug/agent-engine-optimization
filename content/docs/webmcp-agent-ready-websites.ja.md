# エージェント対応 Web サイト用の WebMCP

WebMCP は、エージェントにスクリーンショット、DOM 構造、または壊れやすいボタンのラベルからアクションを推測させるのではなく、ページレベルのツールをブラウザー エージェントに直接公開する方法を Web サイトに提供するため、重要です。 AEO にとって、WebMCP は人間指向の Web ページとマシンが実行可能なアクションの間の橋渡しとなります。

## WebMCP とは何ですか?

Chrome の WebMCP 資料では、WebMCP について、Web サイトがアプリケーション機能の目的を明確にし、ブラウザー エージェントに構造化された機能を提供するためのブラウザー側の方法として説明しています。 Chrome のガイドでは、WebMCP が MCP の代替ではないことが明示されています。どちらも異なる問題を解決します。

一次情報源:

- [Chrome: WebMCP と MCP を使用する場合](https://developer.chrome.com/blog/webmcp-mcp-usage)
- [Chrome Lighthouse: 登録済み WebMCP ツール](https://developer.chrome.com/docs/lighthouse/agentic-browsing/registered-webmcp-tools)
- [WebMCP 人形遣いガイド](https://pptr.dev/guides/webmcp)

## WebMCP と MCP の比較

|質問 | PCM | WebMCP |
|
---|
---|
---|
|どこで運用されていますか? |サーバー環境またはツール |ブラウザページのコンテキスト |
|主な仕事 |エージェントをツール、API、データ、ワークフローに接続する |ページがサポートするアクションをブラウザーエージェントに伝える |
|良いこと |バックエンド アクション、データ アクセス、ビジネス ツール |フォーム、ページアクション、支払い手順、予約 |
|発見方法 | MCP サーバーがツールを発表 |ページは宣言的に、または JavaScript を使用してツールを登録します。
| AEO機能 | API とサービスの実行層 | Web インターフェイスの実行層 |

[MCP vs API ガイド] (/docs/mcp-vs-api-for-agents/) では、サーバー側の MCP について説明しています。 WebMCP は、ブラウザーが依然としてユーザー ジャーニーの一部であるサイトにページ レベルのレイヤーを追加します。

## WebMCP が AEO を変更する理由

多くの Web サイトには、予約の予約、カートへの製品の追加、見積もりのリクエスト、在庫のフィルタリング、サポート チケットの送信などのアクションがすでに用意されています。問題は、これらのアクションは通常、UI 要素としてのみ表示されることです。

ブラウザ エージェントは人間のようにページを操作しようとすることができますが、それは脆弱です。ラベルが変わります。マナーが現れます。フィールドは非表示になります。 A/B テストでは DOM が変更されます。 WebMCP は、サイトにアクションを直接示す方法を提供します。このフォームは予約を予約します。これらのフィールドは必須であり、ユーザーが予約を希望する場合はこのアクションを使用する必要があります。

これは、ボタン ラベルだけよりも強力な [実行層](/es/docs/execution-layer/) シグナルです。

## 実装パターン

Chrome の Lighthouse ドキュメントには、ツールはフォームの宣言属性を使用して、または「navigator.modelContext.registerTool」のような命令型 JavaScript API を使用して登録できると記載されています。

|パターン |に最適 |例 |
|
---|
---|
---|
|宣言型ツール |明確なアクションを持つ既存のフォーム | `予約予約`、`見積もり依頼` ||必須の登録 |動的なアプリケーションと複雑なアクション |在庫検索、カートの突然変異 |
| MCPサーバー |ブラウザに依存しないバックエンド ワークフロー |顧客検索、請求書作成 |
|ハイブリッド | API と Web UI アクションを備えたサイト | e-コマース、予約、SaaS パネル |

ほとんどのサイトでは、最初の有効な WebMCP アクションは、リードのリクエスト、予約の問い合わせ、サポート パス、または製品フィルターなどの低リスクのフォームである必要があります。

## 優れた WebMCP ツールとは何でしょうか?ツール名はアクション指向で安定したものにする必要があります。説明では、それをいつ使用するか、どのような結果が期待されるかをエージェントに伝える必要があります。

弱い：

```text
toolname="submit"
tooldescription="Submits the form"
```

より良い:

```text
toolname="request_audit"
tooldescription="Request an AEO readiness audit for a website. Use when the user wants a human review of AI search, crawler, and execution-layer readiness."
```

エージェントには、意図、必要な入力、および期待される結果が必要です。 [エージェント ボタンとアクションのアウトライン ガイド] (/docs/agent-buttons-action-schema/) では、WebMCP を超えたアクション ステートメントについても同じ原則が説明されています。

## SEO と AEO への影響

WebMCP は従来のランキング要素ではありません。これはランキングへの近道ではなく、実行可能性の兆候として扱います。

便利な SEO/AEO シーケンスは次のとおりです。

1. ページをクロール可能およびインデックス可能にします。
2. 明確なコンテンツ、タイトル、概要、内部リンクを追加します。
3. 検出ファイルを [llms.txt](/es/docs/programming-llms-txt/) として公開します。
4. 高価値株用の WebMCP ツールを追加します。
5. Lighthouse エージェントのナビゲーション監査を試してください。
6. フォームの完了とエージェントによってトリガーされるイベントを監視します。

ページのコンテンツが弱い場合、WebMCP はそれを修正しません。エージェントがページを理解したら、アクションを起こすのに役立ちます。

## リスク

確認と許可なしに機密性の高いアクションを公開しないでください。ネイティブ ブラウザ アクションでは、適切な同意なしに、エージェントが請求内容を変更したり、データを削除したり、購入を完了したりすることを許可してはなりません。

優れた手すりには次のようなものがあります。

- 明確なツールの説明
- 高リスクアクションの明示的な確認
- サーバー側の検証
- 監査ログ
- レート制限
- 規制されたタスクに対する人によるレビュー

## よくある質問

### WebMCP は MCP と同じですか?

いいえ、MCP はエージェントをツールやシステムと接続します。 WebMCP は、Web サイトがページレベルのアクションをブラウザー エージェントに公開するのに役立ちます。

### すべての Web サイトは WebMCP を実装する必要がありますか?

まだ。エージェントがフォーム、予約、フィルター、またはその他のブラウザー操作を完了する必要があるサイトから始めます。

### WebMCP はランキングを向上させますか?

WebMCP がランキング要因であるという公式な主張はありません。その価値は従来のSEO順位ではなく、エージェントの行動力です。

### 最初に何を実装すればよいでしょうか?

見積もりの​​リクエスト、予約、サポート パスの開設など、リスクの低いアクションから始めてください。

### WebMCP をテストするにはどうすればよいですか?

Lighthouse の Chrome ブラウジング エージェント監査と、登録された WebMCP ツールを一覧表示できるブラウザ ツールを使用します。
