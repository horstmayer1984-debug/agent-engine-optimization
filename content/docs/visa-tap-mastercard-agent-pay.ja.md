# Visa TAP と Mastercard Agent Pay: AI エージェント向けのカード ネットワーク支払い

Visa TAP と Mastercard Agent Pay は、既存のカード ネットワーク インフラストラクチャにエージェント支払いを追加します。 x402とは機能が異なります。彼らは、暗号ネイティブの契約よりも、信頼できるエージェント、トークン化された資格情報、販売者の信頼、および使い慣れた支払い経路内の不正行為の制御に重点を置いています。

ほとんどの販売業者はすでにカード ネットワーク、アクワイアラー、詐欺計画、トークン化を運用しているため、この区別は重要です。誰もがステーブルコインでエージェントへの支払いを開始するわけではありません。

## 簡単な比較

|基準 | Visa Trusted Agent プロトコル | Mastercard 代理店支払い |
|
---|
---|
---|
|主な目的 |販売者が正規の AI エージェントを識別し、認証情報のコンテキストを渡すのを支援 | Mastercard のトークン化と支払いネットワーク機能を通じてエージェントのトランザクションを可能にする |
|中心機構 |エージェント ID と信頼できるリクエスト コンテキスト | Mastercard Agentic トークンとトークン化された支払い資格情報 |
|ベストフィット | Merchant Trust、エージェント検証、支払いコンテキスト |カードベースのエージェント購入とネットワーク制御の認証 |
|暗号プロトコルとの関係 | x402 の直接の代替品ではなく、補完的なものです。カードレーンに重点を置いた補完的なもの |
|商用導入ルート |加工業者と取引の既存のエコシステム |既存の Mastercard の受け入れとトークンのインフラストラクチャ |

## Visa TAP は何をするのですか?

Visa の信頼できるエージェント プロトコルは、販売者の信頼に関する質問、つまりこのリクエストは正規の AI エージェントからのものか? を中心に設計されています。販売者は、不審なボット トラフィックとは異なる扱いをすることができますか?

Visa Developer の概要では、販売者とのやり取りを通じてエージェント関連のコンテキストと支払い資格情報を伝達するフローについて説明します。エージェントは商品を閲覧、比較し、オプションを選択し、支払い詳細を送信できるため、このプロトコルはチェックアウト前およびチェックアウト中に重要です。

AEO の場合、TAP は新しい最適化サーフェスを指します。販売者は、信頼できるエージェントをスクレイピングボットや詐欺の自動化から区別しながら、エージェントに優しい取引ルートを公開する必要がある場合があります。

## Mastercard Agent Pay の機能

Mastercard Agent Pay は、Mastercard のトークン化インフラストラクチャをエージェント コマースに適用します。 Mastercardは、Agenticトークンは、非接触型モバイル決済、カード・オン・ファイル・コマース、決済アクセスキー、プログラム可能な決済にすでに使用されているトークン化機能に基づいて構築されていると説明しています。

中心的なアイデアは、カード ネットワークを置き換えることではありません。その目標は、AI エージェントがユーザーに代わって動作するときにカード支払いを利用できるようにすることです。これは、一般的な企業や小売店での導入にとって重要です。多くの企業は、既存のカードの受け入れ、紛争、不正行為、および報告システムに適合するルートを好みます。

## x402 との違いは何ですか

x402 は Web ネイティブであり、多くの場合ステーブルコインを対象としています。 Visa TAP と Mastercard Agent Pay はカード ネットワークを対象としています。|使用例 |最適なファーストフィット |
|
---|
---|
| API リクエストごとの支払い | x402 |
| AIエージェントが量販店でカード決済で購入 | Visa TAP または Mastercard 代理店支払い |
|販売者は支払い前に代理店の身元を証明するサインを求めています |タップビザ |
|カード ネットワークは代理店購入のための特定の認証情報を必要としています | Mastercard 代理店支払い |
| MCP ツールは少額の呼び出し手数料を請求します | x402 または MPP |

## カードエージェントによる支払いが重要な理由

カードは、多くの電子商取引において依然としてデフォルトの支払い方法です。ステーブルコインのプロトコルが急速に成長したとしても、ほとんどの販売業者はすでにカード処理装置、不正行為のルール、チャージバックのワークフロー、会計システムを導入しています。

代理店への支払いはその現実に適応する必要があります。

カードベースのエージェント支払いレイヤーは、次の解決に役立ちます。

1. ユーザー資格情報の保護
2. 支出制限
3. エージェントトラフィックにおける販売者の信頼
4. 不正行為の分類
5. 紛争管理
6. トークン化された支払い
7. 既存の取得者および処理者との互換性

## AEO が販売者に与える影響

Visa TAP または Mastercard Agent Pay の準備をしている販売者は、支払いコードだけから始めるべきではありません。サイトには、エージェントにとってクリーンで読みやすいビジネス画面が必要です。

役立つ準備手順:

1. 構造化製品データを公開します。
2.価格、バリエーション、在庫状況、ポリシーを追跡可能にする
3. 文書の撤回の要件
4. 信頼できるエージェントのトラフィックを一般的なボットのトラフィックから分離する
5. 支出と認可の制限を定義します。
6. 購入後の検証エンドポイントを提供する
7. エージェント マニフェストまたは llms.txt から支払い機能をリンクする

これにより、支払い処理だけでなく、支払いの準備が [エージェント エンジンの最適化](/es/docs/what-is-aeo/) に接続されます。

## よくある質問

**Visa TAP は支払いプロトコルですか?**
Visa TAP は、支払い認証情報の管理と販売者の検証に関連する、AI コマースのための信頼およびエージェント識別フレームワークとして最もよく理解されています。

**Mastercard Agent Pay は暗号通貨に基づいていますか?**
Mastercard Agent Pay は、Mastercard のネットワーク機能とトークン化に重点を置いています。これは、x402 のような暗号ネイティブ HTTP 決済プロトコルと同じカテゴリではありません。

**加盟店には x402 およびカード ネットワーク システムが必要ですか?**
そうする人もいます。小売店の支払いではカード ネットワーク システムを使用できますが、有料 API、データ フィード、または MCP ツールでは x402 または MPP を使用できます。

**小売業者は最初に何をすべきですか?**小売業者は、支払いレイヤーを選択する前に、商品データ、ポリシー、支払い要件、購入後のフローをエージェントが読み取れるようにする必要があります。
